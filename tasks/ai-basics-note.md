
AI 基础概念学习笔记

这篇笔记是我用自己的话整理的 AI 基础概念，主要是为了帮助自己建立后续学习 Agent、workflow、AI coding 和工具调用时的共同语言。


1. LLM｜大语言模型

一句话解释

LLM 是一种可以理解和生成自然语言的 AI 模型，它通过大量文本训练，学会根据上下文预测和生成内容。

具体例子

当我让 ChatGPT 帮我解释 Web3 概念、写学习笔记、修改 Markdown 格式时，它就是在使用大语言模型理解我的需求，并生成相应的文本。

常见误区或使用边界

LLM 不是“真正知道一切”的人，它更像是一个基于大量语言模式进行推理和生成的工具。它可能说得很流畅，但内容不一定总是准确，所以涉及代码、财务、法律、医学或重要学习任务时，需要自己复核。



2. Prompt｜提示词

一句话解释

Prompt 是我给 AI 的任务说明，它决定了模型要做什么、按什么格式输出、用什么语气表达。

具体例子

如果我只说“帮我写笔记”，AI 可能会写得很泛。但如果我说“请用 GitHub Markdown 格式，整理 6 个 AI 基础概念，每个概念包含解释、例子和误区”，输出就会更接近我的真实需求。

常见误区或使用边界

Prompt 不是越长越好，而是越清楚越好。好的 prompt 应该说明目标、背景、格式、限制和输出标准。如果任务本身没有想清楚，prompt 再复杂也可能得到不理想的结果。



3. Context Window｜上下文窗口

一句话解释

Context window 指模型一次能够“看到”和参考的信息范围，包括我输入的问题、之前的对话、文件内容或工具返回的信息。

具体例子

如果我在同一个对话里连续讨论几个不同的问题，模型可以根据前面的内容继续回答。但如果上下文太长，模型可能会遗漏较早的信息。

常见误区或使用边界

上下文窗口不是无限的。不能默认 AI 永远记得所有细节，尤其是长对话或跨平台任务中，最好把关键背景重新说清楚。例如文件路径、任务要求、输出格式、都应该在新任务里重新说明。



4. Workflow｜工作流

一句话解释

Workflow 是把一个复杂任务拆成多个可执行步骤，并按照一定顺序完成的流程。

具体例子

我用 AI 辅助完成 GitHub 打卡，其实就是一个 workflow：先理解任务要求，再生成 Markdown 内容，然后复制到 GitHub 新建文件，最后 commit 提交。

常见误区或使用边界

Workflow 不是简单地让 AI 一次性完成所有事情。更可靠的做法是把任务拆开：先让 AI 生成内容，再检查格式，再提交文件。这样每一步都能确认，出错时也更容易定位问题。



5. Agent｜智能体

一句话解释

Agent 是一种更主动的 AI 系统，它不只是回答问题，还能围绕目标进行规划、调用工具、执行步骤并根据结果调整行动。

具体例子

如果我让普通聊天模型“解释什么是测试网”，它会给我一段解释。但如果是一个学习 Agent，它可能会先阅读我的学习计划，再生成今日任务，帮我创建 daily 文件，检查 GitHub 提交状态，并提示我下一步该做什么。

常见误区或使用边界

Agent 不等于完全自动化，也不代表可以放心让它做所有决定。特别是涉及写文件、提交代码、转账、删除内容、发送邮件等操作时，人仍然需要确认。Agent 越有执行能力，越需要边界和权限控制。



6. Tool Use｜工具调用

一句话解释

Tool use 指 AI 在回答问题之外，还可以调用外部工具完成任务，比如搜索网页、读文件、运行代码、修改文档或操作日历。

具体例子

Claude Code 可以读取我的项目文件，理解 daily 文件夹里的历史记录，然后帮我创建新的 Markdown 文件。这就不是单纯聊天，而是 AI 在调用本地文件和代码工具完成任务。

常见误区或使用边界

工具调用让 AI 更强，但也让风险更高。比如 AI 可以写入文件，也可能误改文件；可以运行命令，也可能执行不合适的操作。所以在允许工具调用前，要看清楚它准备做什么，尤其是涉及删除、覆盖、上传、提交和密钥配置时。



7. AI Coding｜AI 辅助编程

一句话解释

AI coding 是用 AI 帮助写代码、解释代码、调试错误、生成项目结构或完成开发任务。

具体例子

我使用 Claude Code 让它根据已有的 daily 文件生成新的学习记录，虽然这不是传统意义上的写程序，但它仍然属于 AI coding 的一种使用方式：AI 在项目目录里读取、创建和修改文件。

常见误区或使用边界

AI coding 不代表我完全不用理解代码和命令。比如终端里哪些命令应该在普通 zsh 中输入，哪些内容应该在 Claude Code 交互界面输入，如果分不清，就会不断报错。所以使用 AI coding 时，仍然要理解最基本的文件路径、终端命令和执行环境。



8. Guardrails｜安全边界

一句话解释

Guardrails 是给 AI 系统设置的规则和限制，用来降低错误输出、越权操作或不安全行为的风险。

具体例子

在我的学习任务里，一个重要 guardrail 是：不要把 API Key、token、.env 文件、真实私钥或助记词提交到 GitHub。即使 AI 帮我生成笔记，也应该明确提醒它不要写入敏感信息。

常见误区或使用边界

Guardrails 不是绝对保险。即使设置了规则，AI 仍可能因为上下文不完整或用户指令不清楚而犯错。所以安全边界需要模型、工具和人共同维护，不能只依赖 AI 自己判断。



9. Tracing｜过程追踪

一句话解释

Tracing 是记录和观察 AI 系统执行过程的方法，用来帮助我们知道它做了什么、调用了什么工具、在哪一步出错。

具体例子

如果 Claude Code 生成文件失败，我需要看它是在登录失败、API 连接失败、文件路径错误，还是写入权限失败。这个排查过程就是一种简单的 tracing。

常见误区或使用边界

很多新手只看最终结果，不看中间过程。但使用 Agent 或 AI coding 时，中间过程很重要。只有知道错误发生在哪一步，才能判断是 prompt 问题、工具问题、API 问题，还是自己把命令输错了位置。



10. Human-in-the-loop｜人在回路中

一句话解释

Human-in-the-loop 指在 AI 执行任务的过程中，人仍然参与检查、确认和决策，而不是完全交给 AI 自动完成。

具体例子

AI 可以帮我写 GitHub Markdown，但最终我需要自己检查内容是不是符合课程要求，有没有泄露 API Key，有没有把没有学习过的内容写成“已经完成”。

常见误区或使用边界

使用 AI 不等于放弃自己的判断。尤其是在学习、写作、代码提交、资产操作和账号配置中，人应该负责最后确认。AI 可以提高效率，但不能替我承担理解和责任。



我如何使用 AI 辅助并进行复核

这篇笔记使用了 AI 辅助整理，但我没有直接把 AI 输出当成最终答案，而是按照自己的学习场景进行了复核和改写。

我的复核方式主要包括：

1. 检查每个概念是否能帮助我理解后续的 Agent、workflow、AI coding 和工具调用。
2. 把解释改成更接近自己能理解的语言，而不是只保留抽象定义。
3. 增加和自己学习场景相关的例子，例如 Claude Code、GitHub Markdown、daily 文件和 API 配置。
4. 检查是否包含误区或边界说明，避免只写概念不写风险。
5. 确认没有包含 API Key、token、.env 文件或其他敏感信息。



今日学习总结

通过整理这些概念，我发现 AI 工具并不是一个单一的“聊天机器人”，而是可以拆成模型、提示词、上下文、工作流、工具调用和安全边界等多个部分。

后续继续学习 Agent 和 AI coding 时，我需要重点理解两件事：

1. AI 能帮我完成很多重复性和结构化任务，但我必须清楚任务目标和执行边界。
2. 越是能调用工具、修改文件、执行命令的 AI，越需要人来检查和确认。

对我来说，学习 AI 的关键不是只会问问题，而是逐渐学会设计任务、拆解流程、验证结果，并在安全边界内使用工具。
