# Week 2 Module B｜Agent Commerce Payment Flow 设计

## 1. 文档目标

本文档基于 Week 2 Module B：Payment / Commerce / Settlement 完成。

本模块的核心不是简单讨论“Agent 能不能转账”，而是分析当 AI Agent 参与经济活动时，如何把报价、预算、授权、执行、交付、验收、托管、争议处理、收据和结算串成一个可控闭环。

我选择的场景是：

# AI Agent 帮用户购买链上地址风险评分服务，并在验收后完成付款

这个场景可以理解为一个最小版的 Agent Commerce Flow：用户不直接自己找 API 服务，而是让 Agent 帮忙选择一个链上地址风险评分服务，完成报价、预算授权、服务调用、结果验收和付款结算。

---

## 2. 场景选择

## 2.1 场景名称

**Agent Risk Score Purchase Flow**

中文名：

**AI Agent 代用户购买链上地址风险评分服务**

---

## 2.2 场景描述

用户想判断某个钱包地址是否存在风险，例如：

- 是否与可疑地址交互过；
- 是否参与过钓鱼合约；
- 是否有异常资金流；
- 是否曾被标记为高风险地址；
- 是否适合继续交互或转账。

用户自己不想注册多个 API 服务，也不想手动比较价格和接口文档。因此，用户委托一个 AI Agent 帮自己完成：

1. 理解需求；
2. 寻找合适的风险评分 API 服务；
3. 获取报价；
4. 在预算范围内选择服务；
5. 调用服务；
6. 验收返回结果；
7. 如果结果有效，则付款；
8. 如果结果无效，则退款或进入争议流程。

这个场景适合作为 Payment / Commerce / Settlement 的最小项目，因为它既有真实服务购买，也有交付结果、验收标准和结算动作。

---

## 3. 参与角色拆解

| 角色 | 对应身份 | 职责 |
|---|---|---|
| 下单方 Client | 用户 / 用户的委托 Agent | 提出任务需求，设置预算，确认是否付款 |
| 执行方 Provider | 链上地址风险评分 API 服务商 | 提供地址风险评分结果 |
| 验收方 Evaluator | 用户指定的 AI Evaluator / 规则验证器 / 人工确认 | 判断服务结果是否符合验收标准 |
| 付款方 Payer | 用户钱包 / 用户授权预算账户 | 在预算范围内支付服务费用 |
| 仲裁方 Arbitrator | 平台仲裁规则 / 人工仲裁 / 多签委员会 | 在服务结果存在争议时判断退款、重做或释放付款 |
| 记录方 Recorder | 链上合约 / 结算日志 / GitHub 文档 | 记录任务、预算、交付、验收和结算结果 |

---

## 4. 为什么这个场景适合 AI × Web3？

## 4.1 如果没有 AI，为什么难以解决？

如果没有 AI，用户需要自己完成很多复杂操作：

- 搜索不同的链上数据 API；
- 阅读接口文档；
- 比较价格；
- 判断服务是否适合当前需求；
- 手动调用 API；
- 检查返回字段是否完整；
- 自己写验收标准；
- 整理付款和争议材料。

对于 Web3 初学者或非技术用户来说，这个流程非常重。AI 的价值在于理解用户需求、拆解任务、生成验收标准、比较服务、解释结果，并在争议时整理证据。

---

## 4.2 如果没有 Web3，为什么缺一块？

如果没有 Web3，这个场景可以做成一个普通 API 购买平台，但会缺少几个关键能力：

- 无法形成开放、可验证的付款记录；
- 难以实现无信任托管；
- 结算依赖中心化平台；
- 服务商和用户之间的争议缺少透明记录；
- Agent 的预算授权和付款边界不清晰；
- 用户很难验证“任务是否已付款、是否已交付、是否已退款”。

Web3 在这里提供的不是“普通支付按钮”，而是托管、状态记录、结算证明和可验证交易历史。

---

## 5. 最小 Payment / Commerce Flow

本设计中的最小闭环包含：

> 报价 → 预算授权 → 执行 → 交付 → 验收 → 付款 / 退款 / 争议 → 记录证明

---

```text
用户提出需求
   ↓
AI Agent 拆解任务
   ↓
服务商报价
   ↓

   ↓
资金进入托管合约
   ↓
服务商执行任务
   ↓

   ↓

   ↓
验收通过 → 释放付款

   ↓
生成链上记录与收据
