## Feedback 1：Vibe Coding 章节，按"学习者来源"分两条入门路径

- 页面：https://aiweb3.school/zh/handbook/ai/vibe-coding/
- 类型：结构建议（章节开头加分流）
- 来源 daily：daily/2026-06-01.md（Day 15）

### 问题

Vibe Coding 这一节现在按「概念 → 工具 → 实践」线性组织，但**「我是谁、我应该从哪里切入」这个分流没显式给出**。

我自己是「AI 有基础 / Web3 新手 / 无传统开发经验」的学习者，第一次读时最大的卡点不是"Vibe Coding 是什么"（这个 1 分钟读完），是**"我现在装 Cursor 还是 Claude Code？哪个先跑通 Hello World？"**。Handbook 默认读者已经知道这两个工具的区别，但实际上从 AI 侧 / 产品侧 / Web3 侧来的学习者各自的最优起点完全不同：

| 学习者来源 | 最容易卡的地方 | 最优起点 |
| --- | --- | --- |
| AI 有基础 / Web3 新手 / 无代码 | 工具选择 + 第一段代码该写什么 | Claude Code（CLI + 终端友好）+ 学习仓库的 daily note 自动化 |
| 传统开发者 / 转 Web3 | "Vibe Coding 跟我已经在用的 Copilot 有什么区别" | Cursor（IDE 内 diff 视图最熟悉）+ 改自己已有的项目 |
| 产品 / 设计 / 非技术 | "我能不能不写代码就跑起来" | Cursor Compose（自然语言生成 → 一键运行）+ 改一个现成模板 |

Day 15 我自己花了大半天才把"该装哪个 + 第一个 Hello World 写什么"摸清楚 — 而且事后回看，**这个分流是 Handbook 完全可以提前给出的**，没必要让每个学习者重复踩一遍。

### 建议改法

在 Vibe Coding 章节最开头加一个「**入门路径分流**」小区块，包含：

1. 一张 3 行的来源 → 起点表（如上）。
2. 每条路径配一个最小 Hello World 任务（比如 AI 侧：用 Claude Code 写一段 daily note 模板生成器；传统开发者：用 Cursor 给已有项目加一个 README；非技术：用 Cursor Compose 改一个 Next.js 模板的标题色）。
3. 一句 anchor：「**Vibe Coding 不是『换一种工具』，是『换一种工作流』 — 选起点比选工具更重要**」。

### 补充上下文

我作为 AI × Web3 School 的学员，整周（Day 15 → Day 18）一直在用 Claude Code，Day 18 让 Claude Code 帮我写 ethers.js 读 Sepolia 余额脚本时这个判断被验证 — 在 CLI + 终端 + 学习仓库这种场景下，Claude Code 的默认值跟我的工作流匹配度极高，跟当时该不该装 Cursor 没关系。这条经验对其它「AI 有基础 / Web3 新手 / 无代码」的同学完全可以复用，但 Handbook 现在没显式说，新人只能每个都试一遍。

## Feedback 2：MCP 章节，把「写入工具权限升级」+「MCP × Web3 钱包边界」用一个具体场景讲清

- 页面：https://aiweb3.school/zh/handbook/ai/mcp/
- 类型：结构建议（新增小节 + 配场景图）
- 来源 daily：daily/2026-06-02.md（Day 16）+ daily/2026-06-03.md（Day 17）

### 问题

MCP 章节有两处现在留白比较大，**而这两处恰好是 AI × Web3 学员最关心的那条主线**：

**留白 1：写入工具权限升级方案没有具体样例**

「最小实践」一节给了只读 server 的清单（search_docs + get_file），但**「写入工具的权限升级方案」只用了一句话带过**：「设计……什么时候需要用户确认，如何撤销授权，如何审计」。学习者第一次读时无法判断「升级到写入」具体长什么样 — 是弹个浏览器确认窗口？还是 CLI 提示 y/N？还是必须接一个 OAuth-like 的授权服务器？

**留白 2：MCP 和 Web3 钱包 / Agent Wallet 在权限上怎么分工，没具体场景**

章节里讲了 MCP 与权限系统的关系（「权限要在协议外也成立」），但**没有给一个具体场景说明 MCP 和 Web3 钱包在权限上怎么分工**。我作为「AI 基础好 + Web3 新手」的学习者，第一次读时最想知道的是：**如果一个 MCP server 暴露了 `sendTransaction` 工具，Permission 边界到底由 MCP 守还是钱包守？两者怎么不打架？**

这两处其实是同一个问题在两个层级的体现 — **MCP 协议本身只描述「能力可被调用」，真正的安全边界在协议外**。这条原则 Handbook 在节中讲过，但具体怎么落地（特别是落到 Web3 场景）没有展开。

### 建议改法

在 MCP 章节末尾「在 AI × Web3 中的位置」那一段后面，新增两个具体场景：

**场景 A：写入工具权限升级清单**

给一个具体的「写入工具权限升级清单」例子，至少包含 5 条：

| 升级项 | 具体形态 |
| --- | --- |
| 单次会话授权 | 客户端启动时读 config，会话内不再问；客户端关闭 = 授权失效 |
| 写入前 dry-run | server 必须提供 `dry_run` 模式，把"将要做什么"返回给 client，让模型 / 用户 review |
| 写入操作 = 显式确认 | client 弹窗 / CLI 提示，必须确认目标路径 + 内容摘要 |
| 写入日志单独留存 | 不写到 stdout，写到独立 log file，方便审计 |
| 关闭客户端 = 撤销 | 不允许 server 在 client 关闭后继续保留任何 session 状态 |

**场景 B：MCP × Web3 钱包的权限分工**

给一个具体例子（Agent 用 MCP 调用一个生成交易草稿的 server，但最终签名走 Smart Account + Session Key）：

```
用户 → Agent → MCP `prepareSendTx` → 返回未签名 tx 草稿
                                ↓
                    Smart Account + Session Key 检查 policy
                                ↓
                          满足 policy → 自动签 + 广播
                          不满足 → Human Check 拦截
```

配一句 anchor：

> **MCP 守"工具发现 + 调用格式 + 调用日志"，Web3 账户系统守"签名 + 执行边界 + 撤销路径"。两者职责分离，谁都不替代谁。**

### 补充上下文

这条 feedback 的实操背景是 — Day 16 我给 Claude Code 接了最小 filesystem MCP（只读 + 白名单），Day 18 用 Claude Code 写 ethers.js 读 Sepolia 余额时复用了这个 MCP。**整个 Week 3 我都在脑子里反复构造"如果是写入会怎样 / 如果是链上动作会怎样"这两个推演**，但 Handbook 没给参照系。如果章节里能直接给出这两个场景，我可以省掉 2 天的自己摸索时间。

## Feedback 3：Frameworks 章节，加一段「什么时候不该上框架」+「Web3 交叉场景的额外约束」

- 页面：https://aiweb3.school/zh/handbook/ai/frameworks/
- 类型：结构建议（节末新增 2 个小节）
- 来源 daily：daily/2026-06-03.md（Day 17）+ daily/2026-06-04.md（Day 18，ethers.js 实操验证）

### 问题

Frameworks 章节把 LangChain / LangGraph / Agents SDK 各自定位讲了，但**两处特别重要的「反向引导」没有**：

**留白 1：「什么时候不该上框架」没单独拎出来**

我作为「AI 有基础 / 无代码」的学习者，第一次读时**最容易错的不是选错框架，是太早上框架** — 一段一次性的 50 行脚本被裹一层 LangChain 之后，调试和阅读都变难，反而拖累 Vibe Coding 工作流。Day 17 我自己花了一晚才得出这个判断（按 Frameworks 5 个共同问题逐条对照「这段脚本是不是真需要框架」），Day 18 用 ethers.js 读 Sepolia 余额脚本时这个判断**得到了实操验证** — 不上框架，4 行核心代码，10 分钟跑通。如果当时被 LangChain 裹一层，至少多 1 小时配置 + 调试。

**留白 2：Web3 交叉场景的额外约束没系统讲**

章节讲了框架在 AI 应用层的位置，但**没系统讲框架在「AI × Web3」交叉场景的额外约束** — 链上动作有不可逆 / gas / 不可重试的硬约束，这跟纯链下 Agent 编排的失败语义完全不同。学习者读完之后容易以为「LangGraph 的 Checkpointer = 一切失败都能从断点恢复」，但 checkpoint 是链下状态，**链上 tx 已经上链了不可能回滚**。

这两个留白对 AI × Web3 学员特别要紧 — Week 5 / 6 进 AI × Web3 Bridge 时，每个人都会撞到「框架的 retry / checkpoint / interrupt 在链上意味着什么」这个问题。

### 建议改法

**新增小节 A：「3 个还不该上框架的信号」**

在 Frameworks 节末尾加一段，明确列出 3 个反向信号：

| 信号 | 解释 |
| --- | --- |
| **单步无状态** | 一次调用、无分支、无循环 — 上框架是负杠杆 |
| **一次性脚本** | 跑一次就完事，不会被复用 / 不会被多人维护 |
| **还没跑过裸 SDK 最小闭环** | 先用裸 SDK 把"这件事到底有多复杂"显性化，再判断要不要上框架 |

配一句 anchor：

> **框架在「同一段逻辑要重复 + 多人维护 + 要能调试」时才赚回它的复杂度。如果三条都不满足，不上框架就是最优解。**

**新增小节 B：「框架在 Web3 场景的 3 条额外考量」**

| 考量 | 具体形态 |
| --- | --- |
| **链上动作要 dry-run** | 任何写交易前必须先 `eth_call` 模拟，把「会失去什么 / 得到什么」返回给用户 review |
| **Checkpoint 要带 tx hash 列表** | LangGraph / Agents SDK 的 checkpoint 是链下状态，必须额外维护一个「这个 checkpoint 之前已经发出了哪些链上 tx」的列表，否则恢复时会重复发交易 |
| **中断点要落在「签名前」而不是「签名后」** | Human Check / Interrupt 必须在 `eth_sendTransaction` 之前发生，签完之后再 interrupt = 回不来了 |

预告这些会在 AI × Web3 Bridge 那一章详细展开。

### 补充上下文

这条 feedback 的实操背景是 Day 17 + Day 18 两天连贯：Day 17 我读完 Frameworks 之后写下「这段 ethers.js 脚本不上框架」的预判，Day 18 实操验证这个预判正确（4 行核心代码 + 10 分钟跑通）。这个**「预判 → 实操验证」闭环**让我反过来看清 Handbook 留白的具体形状 — 不是 Handbook 写得不好，是「反向引导（什么时候不用 / 在 Web3 多了哪些约束）」这条线没显式拉出来。
