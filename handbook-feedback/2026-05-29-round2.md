## Feedback 1：Account Abstraction 章节，把"核心不是免 gas"提到节首作 anchor

- 页面：https://aiweb3.school/zh/handbook/web3/account-abstraction/
- 类型：结构建议
- 来源 daily：daily/2026-05-28.md（Day 11）

### 问题

从 Web2 / 普通用户视角进 AA 这一节，会被前面 Paymaster / gasless / onboarding demo 的叙事先入为主，**Handbook 自己的关键金句"AA 的核心不是免 gas，而是把账户控制权从单一私钥扩展成可编程规则"被压在了"为什么要学这个"之后**，新学习者很容易错过这条。

我自己在 Day 11 之前对 AA 的理解一直是"AA = gasless + 社交恢复"，直到读到这一句才意识到 gasless 只是 Paymaster 的副产品，**真正解锁的是 Smart Account + Session Key 这条权限轴**。这条认知差对后续做"Agent 上链执行"类项目特别关键 — 如果只停在"AA = 钱包功能升级"视角，会去找现成钱包；但实际要做的是设计**规则**，不是挑钱包。

### 建议改法

1. 把这句话提到章节最开头，加粗作为整节 anchor。
2. Paymaster / gasless 之后单独开小节"AA 的常见误读"，把"AA = gasless"作为第一个误读列出来，配一句反例（"如果你只想要 gasless，不一定需要 AA — 关于 4337 的整套 Bundler/EntryPoint 复杂度对你来说就是过载"）。

### 补充上下文

我是 AI 有基础 / Web3 新手 / 从 AI 侧进 Web3 的学习者。先读了 Bridge · Agent Wallet 那一节才回过头读 AA，这个顺序反而帮我跳过了 gasless 误读，但**正常按目录顺序读的学习者很可能会卡住**。

## Feedback 2：Smart Contract 章节，补一个 "ABI ↔ Agent tool schema" 的小节

- 页面：https://aiweb3.school/zh/handbook/web3/smart-contract/
- 类型：结构建议（新增小节）
- 来源 daily：daily/2026-05-27.md（Day 10）

### 问题

从 AI 侧进来的学习者（特别是有 OpenAI tool / MCP / OpenAPI 经验的）最快建立直觉的方式是 — **把 ABI 看成"链上版的 tool schema"**。

Handbook 现在没把这个类比讲出来，导致 ABI 在文中只是"一段 JSON"，没法立刻接到学习者已有的工具调用心智模型。我在 Day 10 整理对照表时，发现这两件事在结构上几乎一一对应：

| 维度 | Agent 工具调用 | 合约调用 |
| --- | --- | --- |
| 接口描述 | tool schema / OpenAPI | ABI |
| 入口 | API endpoint | 合约地址 + 函数名 |
| 参数 | JSON body | encoded calldata |
| 身份 | API key / Bearer token | 钱包私钥签名 |
| 结果 | response body | 返回值 + event log |
| 成本 | 通常按调用计费或免费 | gas（写）/ 免费（读） |
| 可逆性 | 服务方一般可补偿 / 重试 | 写操作上链后不可逆 |
| 可见性 | 调用记录在服务方那 | 链上全公开 |

链上调用比工具调用多了三层约束：**钱包签名 / gas 与不可逆 / 状态全网可见**。

### 建议改法

在 Smart Contract 节末尾新增一小节"ABI 作为接口契约"，给一段 ERC-20 `transfer(address,uint256)` 的 ABI 片段和一份对应的 Agent tool schema（OpenAPI / MCP）并排展示，配两行解读：

> 函数名 ↔ tool name / 参数类型 ↔ JSON Schema / 返回值 ↔ response schema。
> 链上多出来的三层约束：签名 / gas / 不可逆。

### 补充上下文

这条类比帮我把 Agent 调链上合约从"全新陌生概念"还原成"我已经熟悉的工具调用 + 三层链上约束"。这是从 AI 侧进来的学习者最快建立直觉的入口，目前 Handbook 没显式给出来。

## Feedback 3：Agent Wallet 章节，把 9 节点压缩成一张"接入清单"表格

- 页面：https://aiweb3.school/zh/handbook/bridge/agent-wallet/
- 类型：结构建议（新增表格）
- 来源 daily：daily/2026-05-28.md（Day 11）

### 问题

Agent Wallet 的 9 个知识节点（AA Wallet / Smart Account / Safe / Session Key / Policy / Guard / Simulation / Revocation / Human Check）现在是按"概念字典"组织的，每个节点有自己的小段。

但**在落地视角，这 9 个节点是一组顺序检查项** — "你已经做了 1 → 2 → 3 → ... 没有，缺哪个？"。我自己花了 Day 11 一整天才把它们重排成可用的清单。

### 建议改法

在 Agent Wallet 节末尾加一张"Agent 受限钱包接入清单"表格 — 行是 9 个节点，列是：

| 节点 | 这一步要回答的问题 | 最小可用方案 | 最常见的偷懒做法（反面） |
| --- | --- | --- | --- |
| AA Wallet | 你的账户能不能表达规则？ | 用任何 ERC-4337 兼容的 Smart Account | 把 EOA 私钥直接交给 Agent — ❌ |
| Smart Account | 谁能动 / 哪些操作要确认 / 单日上限 / 何时失效 这几条都写进账户逻辑了吗？ | 用 Safe / Kernel / Biconomy SCA | 只接了 SCA 但没配置 module — ❌ |
| Safe | 高价值场景是否多签 / 多人执行？ | Safe + 1/N 或 M/N 阈值 | 高价值账户单签 — ❌ |
| Session Key | 时间 / 合约 / 方法 / 单笔 / 总额 都限定了吗？ | Rhinestone Smart Sessions / 自定义 module | 只限了过期时间，金额无上限 — ❌ |
| Policy | "Agent 可以做什么"是写成系统能检查的条件，不是法律条款？ | 用结构化字段表达 + 链上 enforce | Policy 写在 README 里 — ❌ |
| Guard | 执行前的确定性拦截层（白名单 / 金额 / 授权检查）有没有？ | Safe Guard / 自定义 hook | Agent 自己判断"该不该" — ❌ |
| Simulation | 签名 / 广播前能不能让人看懂"会失去什么 / 得到什么 / 哪些授权变了"？ | Tenderly / `eth_call` 模拟 | 直接广播让用户事后看 — ❌ |
| Revocation | 撤销路径有几条？（手动 + 自动） | UI 一键 revoke + 自动收紧（连续失败 / 异常 / 到期） | 只有"等过期"一条 — ❌ |
| Human Check | 是否分层（低 / 中 / 高 / 超 policy）？ | 4 档分层 + 中风险展示影响 | 全自动 / 全手动 — ❌ |

这张表能直接被读者复制到自己的项目 README 里当 checklist。

### 补充上下文

Day 11 我的体感是 — Handbook 里 9 个节点的概念都讲清楚了，但**"我现在这条产品线缺哪一个"**这个问题需要学习者自己再做一轮重排。一张统一的清单表能省掉这一步。
