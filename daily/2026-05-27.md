## 关键概念笔记（Handbook · Smart Contract）

### 1. 智能合约 = 部署到链上的一段可执行代码 + 一段持久化状态

- **一句话**：合约不是「应用」，是一段被矿工 / 验证者执行、状态被全网共识保存的程序。
- **简单例子**：一个 `Counter` 合约里有一个 `uint count` 变量和一个 `increment()` 函数。任何人付 gas 调用 `increment()` 都会让全网那一份 `count` 加一。
- **容易误解**：很多人把「合约 == DApp」。其实合约只是规则那一层，前端、索引、钱包提示这些都不是合约本身。Agent 调合约其实只需要 RPC + ABI，不需要前端 DApp。

### 2. 读（call）和写（transaction）是两件不一样的事

- **一句话**：读链上状态不需要钱包签名也不花 gas，写状态必须由一个有私钥的账户签名并支付 gas。
- **简单例子**：调用 `count()` 查当前计数器值是 read call；调用 `increment()` 改值就必须发交易、等出块。
- **容易误解**：以为「看一下余额也要钱包」。读是免费的、随时的；写才有不可逆性，所以也才有「Agent 写操作必须 human-in-the-loop」这种讨论。

### 3. ABI 是合约暴露给外界的「接口契约」

- **一句话**：ABI（Application Binary Interface）描述了合约里有哪些函数、参数类型、返回值类型，是钱包、SDK 和 Agent 调合约的入口。
- **简单例子**：一个 `transfer(address to, uint256 amount)` 在 ABI 里就是一段 JSON，写明函数名、两个参数和它们的类型。
- **容易误解**：把 ABI 当成「文档」。它其实是机器读的格式，Agent 拿到 ABI 就能像拿到 OpenAPI / MCP tool schema 一样去推断「这个合约能做什么」。

### 4. 合约一旦部署，默认是不可改的

- **一句话**：除非合约自己设计了升级机制（proxy 模式等），否则部署到链上的字节码就是最终版本，bug 也只能靠新部署或迁移来处理。
- **简单例子**：写错了一个权限检查，已经部署上去，攻击面就是真的。
- **容易误解**：以为「线上出问题再 hotfix」。Web2 的发布心智在合约上完全失效，所以 Web3 安全审计、形式化验证、模拟执行才会被反复强调。

## 合约调用 vs Agent 工具调用：同一种动作的两种形态

今天最大的体感是这两件事在结构上几乎一一对应：

| 维度 | Agent 工具调用 | 合约调用 |
| --- | --- | --- |
| 接口描述 | tool schema / OpenAPI | ABI |
| 入口 | API endpoint | 合约地址 + 函数名 |
| 参数 | JSON body | encoded calldata |
| 身份 | API key / Bearer token | 钱包私钥签名 |
| 结果 | response body | 返回值 + event log |
| 成本 | 通常按调用计费或免费 | gas（写）/ 免费（读） |
| 可逆性 | 服务方一般可以补偿 / 重试 | 写操作上链后不可逆 |
| 可见性 | 调用记录在服务方那 | 链上全公开 |

差别在于链上调用多了三层约束：**钱包签名**（谁在调用）、**gas 和不可逆**（调用是有成本的、写完不能撤）、**状态全网可见**（调用过的事任何人都能查）。

所以「AI Agent + 链上工具」不是把工具列表里多塞一个 RPC endpoint 那么简单。需要重点关注的几件事：

- **权限边界**：Agent 用的钱包能签什么、不能签什么，签名额度和过期时间怎么定。
- **人工确认**：哪些动作可以让 Agent 直接发交易，哪些必须停在「草稿交易」那一步等人确认。
- **日志记录**：链上动作天然可被任何人审计，但 Agent 自己的 reasoning 和工具调用日志要单独留好，便于事后复盘和归因。
- **可验证性**：链是天然的「已发生」证据，但「为什么 Agent 决定这么做」这部分仍然在链下，需要单独的可验证记录方式。

钱包不只是「登录按钮」，合约不只是「DeFi 后端」，这两点在今天看 ABI 的时候第一次有了具体的画面感。

## 我对 AI + Web3 的新理解

- **AI Agent 不只是聊天机器人**。当 Agent 能调外部工具、调链上合约时，它在做的是「执行流程」而不是「回答问题」，这是 Handbook 里反复强调的从「回答」到「执行」的跳变。
- **Web3 不只是钱包和转账**。合约这一层让链变成一个可以被任何人调用的「公共后端」，钱包只是签名入口，转账只是合约最简单的一种调用形态。
- **Agent 与 Web3 结合时，权限、调用边界、人工确认、日志记录、可验证性这五件事是一组**。任何一件缺位都会让闭环不成立 — 不是堆功能，是把这五件事在合适的位置放好。
- **工具调用和链上操作之间需要有安全边界**。具体来讲就是：哪些动作在 Intent 边界内自动放行，哪些必须停下来让人签 — 这条边界 Day 9 的笔记里已经写过，今天用合约视角再确认了一次。
