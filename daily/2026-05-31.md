## 今日实践 · Part 1：Counter 合约 Base Sepolia 部署

### 第 1 步：合约准备

打开 Remix → File → 新建 `Counter.sol` → 拷 Day 13 准备的最小源码：

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract Counter {
    uint256 public count;

    event Incremented(address indexed by, uint256 newCount);
    event Reset(address indexed by);

    function increment() external {
        count += 1;
        emit Incremented(msg.sender, count);
    }

    function reset() external {
        count = 0;
        emit Reset(msg.sender);
    }
}
```

Solidity Compiler → 选 0.8.20 → Compile。**第一次编译成功**这件事本身比看起来重要 — 它意味着 ABI 已经生成，Remix 右侧"Deploy & Run"已经知道这个合约有哪些函数能调。

### 第 2 步：网络和钱包

Deploy & Run → Environment → 选 **Injected Provider - Rabby**。

⚠️ **签 Confirm 之前必须确认 Rabby 顶部网络是 Base Sepolia（链 ID 84532）**。这条 Day 7 的硬规则不变 — 一旦在 Rabby 里默认网络是主网，部署一旦签下去就是主网真合约。

如果余额不够 0.001 ETH 那个量级（部署 + 一两次写大概够用），先去 Base Sepolia faucet 领。

### 第 3 步：部署

点 Deploy → Rabby 弹窗 →
- **看 To** — 应该是空（部署交易 To 字段为空，因为部署是**创建**合约，不是**调用**合约）。
- **看 Network** — Base Sepolia。
- **看 Gas** — 极小量级（几分美元等价的测试 ETH，无价值）。

签 → 等出块 → Remix 控制台显示 contract address → 复制下来。

### 第 4 步：调用 increment

Deployed Contracts 里展开新部署的合约 → 点 `increment` 按钮 → Rabby 再签一次。

这一次签名和部署那次是不一样的 tx：
- 部署：`To` 为空（创建合约）。
- increment：`To` = 部署地址（调用合约的 increment 函数）。

签完之后 Remix 控制台再点 `count` 这个 view 函数 → 返回 1（不签名 / 不花 gas，Day 10 那条 read call 不需要签名的认知今天直接验证）。

### 第 5 步：BaseScan 验证

打开 https://sepolia.basescan.org → 粘贴部署地址 → 应该看到两笔交易：

1. **Contract Creation** — 部署那笔。
2. **Increment** — 调用那笔，Method 列显示 `0xd09de08a` 或 `Increment`（取决于合约是否已 verify）。

点进 increment 那笔 → 验证 Day 7 整理的 7 个字段：

| 字段 | 期望 |
| --- | --- |
| Status | Success |
| Tx Hash | 0x... 64 hex |
| Block | 一个具体区块号 |
| Timestamp | 和我点 Confirm 的时间差几秒 |
| From | 我自己地址 |
| To | 合约部署地址（**不是钱包地址**，BaseScan 会标 Contract） |
| Value | 0 ETH（increment 不转 ETH） |
| Txn Fee | 极小 |

然后点 **Logs** 标签 → 看到一条 `Incremented` 事件：
- `by` (indexed) = 我的地址。
- `newCount` = 1。

这是 Day 10 那条「事件 = ABI 的另一半」第一次落地 — 调用 increment 时合约 emit 的事件，**只有读 Logs 才能看到**。这条对后面用 The Graph 做 subgraph 索引（Day 13 那条线）就是直接的入口。

### 第 6 步：（加分项）合约 Verify

BaseScan 上点 "Verify and Publish" → 选 Solidity Single File → 编译版本 0.8.20 → 把源码贴上去 → MIT license → Verify。

成功后 BaseScan 上能直接 Read Contract / Write Contract，而不用再开 Remix。这一步对 Agent 视角特别有意义 — **Agent 拿到一个 verified 合约地址，等于拿到一份开源 + 已编译 + 可直接调的 tool**。

### 部署练习的 3 个具体收获

1. **"部署"和"调用"是两种 tx，但 BaseScan 上看起来差不多** — 区别在 To 字段：部署 To 为空，调用 To 是合约地址。Day 10 那张「合约调用 vs Agent 工具调用」对照表里"入口"这一行今天有了画面感。
2. **事件不在交易主体里，在 Logs 里** — `Incremented` 事件的字段（by / newCount）只在 Logs tab 看得到。这条对 The Graph subgraph 写法的理解直接接通：subgraph 监听的就是这些 Logs。
3. **第一次自己部合约比想象简单，但**"部完之后这个合约就永远在 Base Sepolia 上了"**这件事的体感和写 Web2 代码完全不同**。即使 Counter 这种零风险合约，部上去之后我也没法 hotfix，只能新部一份 — Day 10 那条「合约一旦部署默认不可改」今天有了具体的体感。

## 今日实践 · Part 3：「能解释」清单自检（向我妈解释体）

按 learning-plan 的 Week 2 里程碑：「能解释钱包 / 签名 / 合约调用的底层流程」。今天用一段话向我妈解释每个概念，讲不清楚的当场标 ❌。

### ✅ Smart Contract（合约）

> "智能合约就是一段代码，被人放到一个公共的电脑网络（区块链）上之后，所有人都能看到它、调用它，但谁都改不了。比如有个合约写着『谁付 1 块钱，就给他记一笔账』，那任何人付完之后这笔账就在网络上永远存着，连写合约的人也删不掉。"

### ✅ Account Abstraction（账户抽象）

> "以前你的钱包就是一把钥匙，丢了就完了；钥匙在谁手里，谁就能花你的钱，没有任何条件可以加。账户抽象就是把这把钥匙换成一道带规则的门 — 比如『每天最多花 5 块』『晚上 10 点之后不能花』『一次超过 100 块要二次确认』，规则写进账户里，钥匙就不再是唯一的安全保障。"

### ✅ Agent Wallet（智能体钱包）

> "AI 越来越能帮你做事，但 AI 是会出错的，不能直接拿你的钱包随便花。所以智能体钱包是给 AI 准备的『限制版钱包』 — AI 只能在你提前设好的范围里花（比如花在某几个网站、单笔不能超过多少、过期就自动失效），任何超出范围的动作必须回到你这边确认。"

### ✅ Agentic Commerce（智能体商业 / 服务采购）

> "想象你让 AI 帮你订机票 — 它要去比价、付款、收确认邮件。但 AI 拿什么钱付？怎么证明它真的买到了？买错了怎么退？智能体商业就是把这一整套流程做成『AI 能自动跑、但每一步都有规则可查 + 可撤销』的标准流程。"

### ✅ DeFi（去中心化金融）

> "传统金融里你存钱、借钱、换汇都要找银行；DeFi 是把存 / 借 / 换这些事写成一段段公开的代码（合约），任何人不用开户就能用。核心不是『没有银行』，而是『每件金融工具都能像乐高一样拼起来』 — 一个借贷合约的产出可以直接喂给一个换汇合约。"

### ✅ Indexing（链上数据索引）

> "区块链像一个巨大的账本，所有交易都按时间顺序写进去，但**翻账本找『过去 30 天里 A 给 B 转了多少次钱』极其慢**。所以有人专门写了一种工具（比如 The Graph），先把账本按问题预先整理好，等你查的时候直接返回。"

### ⚠️ Oracle（预言机）— 半通

> "智能合约只看得到链上发生的事，看不到现实世界（比如 ETH 现在多少美元、世界杯谁赢了）。预言机就是一组人，他们把现实世界的事可信地写到链上，让合约可以根据现实做决定。**但用预言机相当于把『相信现实』这件事外包出去 — 预言机被骗了，所有依赖它的合约一起被骗。**"

⚠️ 标"半通"是因为：向我妈这一段说得通，但**"为什么单源价格 Oracle 容易被闪电贷打偏"这种二级解释还讲不清** — 这条留到 Week 3。

### Week 1 时学的（顺便自检）

- ✅ **钱包**：私钥的管理工具 + 签名入口，不是"装币的口袋"。
- ✅ **签名**：用私钥对一段数据做的密码学证明，证明"这个动作是这个地址同意的"，不是"密码"。
- ✅ **合约调用**：客户端构造一段 calldata（"调哪个函数 + 哪些参数"）→ 钱包签名 → 节点广播 → 矿工打包 → 上链。
