## 关键概念笔记 · Part 1：DeFi 五条主线（Web3 基础）

来源：[Handbook · DeFi](https://aiweb3.school/zh/handbook/web3/defi/)

第一性原理（用我自己的话整理）：

> DeFi 不是「在链上重做银行」，而是把**资产 / 价格 / 借贷 / 风险**这四件事拆成可被任何人调用的合约模块，谁都能拼装、谁都能被拼装。

核心定位：DeFi 的关键不是「无中介」，是**可组合性（composability）** — 一个借贷协议的输出（例如 aToken）可以直接当成另一个 AMM 的输入。这一点对 AI Agent 特别要紧 — Agent 调 DeFi 不是调一个银行 API，是把一连串合约串成一条调用链。

### 五条主线（每条一句话讲清 + 锚点协议）

| 主线 | 一句话 | 锚点协议 | 关键风险 |
| --- | --- | --- | --- |
| **资产（Assets）** | 链上的"钱"分四类：原生资产（ETH）/ 稳定币（USDC、DAI）/ 治理或效用代币（UNI、AAVE）/ 凭证类（aToken、LP token、stETH）| USDC / DAI / stETH | 凭证类资产**不是底层资产本身**，价格脱锚、清算、合约 bug 都会让账面数字瞬间失真 |
| **流动性（Liquidity / AMM）** | 用资金池 + 数学曲线（`x*y=k` 等）替代订单簿，任何人存入两种代币就能让别人按曲线兑换，并按比例分手续费 | Uniswap V3 / Curve | 滑点 / 无常损失 / 池子被砸穿 / MEV 抢跑 |
| **借贷（Lending）** | 抵押一类资产去借另一类，全程**超额抵押 + 可清算**，没有信用评分这件事 | Aave / Compound | 健康因子 / 清算瀑布 / 利率突变 / 抵押品价格脱锚 |
| **衍生品 / 永续（Perps & Derivatives）** | 用合约表达"押方向 + 杠杆 + 资金费率"，标的不是现货而是价格指数 | dYdX / GMX / Hyperliquid | 杠杆爆仓 / 资金费率反向 / Oracle 操纵 |
| **稳定币（Stablecoins）** | 想方设法把价格钉在 1 USD：法币储备型（USDC）/ 加密超额抵押型（DAI）/ 算法型（已多次失败，UST/Luna） | USDC / DAI | 储备透明度 / 抵押品脱锚 / 算法稳定币的死亡螺旋 |

### Handbook 给的三条直觉

读完之后我把它压成三条（我自己的语言）：

1. **DeFi 的最小单元是合约，不是产品** — 一个 Aave 不是一个"App"，是一组合约（pool / lendingPool / oracle / liquidator / aToken），每个都能被别的合约直接调。
2. **价格不是事实，是 Oracle 给的视角** — Oracle 决定了"现在 1 ETH 值多少 USDC"。Oracle 错了，整个清算链都错了；这就是为什么 Day 11 / Day 12 都要把 Oracle 当成独立组件看。
3. **凭证类资产是"被抵押的一张白条"** — 不要把 aToken / stETH 当成"和底层资产等价"。它代表"一份在协议里的权利"，权利能否兑现取决于协议本身仍然健康。

### 我对 DeFi 的两个直觉修正（今天）

#### 修正 1：DeFi 的关键词不是"去中介"，是"可组合"

之前看媒体上 DeFi 的叙事，关键词永远是"无银行 / 抗审查 / 抗通胀"。今天读完 Handbook 重新意识到，**这些是结果，不是核心**。核心是"任何人写的金融合约都能被任何人接着调"，这条性质让 DeFi 在产品层面变成乐高 — 借贷协议可以直接当 AMM 的报价源，AMM 又可以当衍生品的现货底，衍生品的 LP 又可以再被借贷。**这条性质同时也是 Agent 的入口**：Agent 不必和一家"DeFi 公司"集成，它面对的是一组开放接口。

#### 修正 2：稳定币不是"链上美元"，是一组**抵押假设**

之前我会模糊地说"USDC = 美元 / DAI = 美元"。今天 Handbook 里那句**"价格能不能锚住 1 美元，取决于背后的抵押和赎回机制"**重新校准了这个直觉。USDC 的"美元"是 Circle 的法币储备承诺；DAI 的"美元"是一组超额抵押的加密资产；算法稳定币（已多次失败）的"美元"只是一组博弈假设。**Agent 在 DeFi 调用时，"金额"不能简单写"X 美元"**，必须落到具体稳定币 + 它的抵押假设上 — 不然在脱锚那一刻策略就错了。

### "Agent 在 DeFi 动手"的分层 — 我的四档清单

按"风险 × 不可逆程度"排，得到 4 档：

| 档位 | 动作示例 | 是否需要主钱包 Confirm | 必备护栏 |
| --- | --- | --- | --- |
| **A · 只读** | 查健康因子 / 查池子价格 / 查持仓 / 查清算阈值 | ❌ 不需要 — Day 10 已经讲过 read call 不签名不花 gas | 仅需「查询频率」和「数据缓存策略」，不需要钱包护栏 |
| **B · 限定额度小写** | 在白名单合约里做小额 swap、领白名单空投、claim 利息 | ⚠️ Intent 那一签需要主钱包；执行可走 Session Key | Session Key + Policy（合约白名单 + 方法白名单 + 单笔 + 总额度 + 过期 + 链 ID）+ Guard（拒绝越界）+ Simulation（执行前预演）+ Revocation（一键收回） |
| **C · 中风险写** | 借贷开仓 / 调整 LP 仓位 / 跨链桥小额转移 / 永续开小仓 | ✋ 每一笔都需要"模拟 + 用户读懂 + 主钱包确认" | B 的全部 + Human Check 中档（弹窗里展示「会失去什么 / 得到什么 / 哪些授权变了」） |
| **D · 高风险或不可逆** | 大额清算 / 加杠杆 / 跨多协议组合策略 / 任何会触发授权 `approve unlimited` 的动作 | ✋✋ 必须主钱包+二次确认；超 Policy 直接拒绝 | B + C 的全部 + Safe 多签 / 多人执行权拆分 + Revocation 立刻可达 |

### 五条主线在 4 档上的位置

按 DeFi 五条主线挨个套：

- **资产**：转移原生 ETH / 稳定币 → A 查 + B 小额转。**aToken / stETH 这种凭证转移天然算 C** — 凭证背后的协议假设可能在转账瞬间已经变了。
- **流动性（AMM swap）**：报价 / 池子状态 → A。小额 swap → B。改变 LP 仓位 / 集中流动性 V3 区间 → C。
- **借贷**：查健康因子 / 利率 → A。还款 / 小额提取 → B。**开仓 / 加杠杆 / 改抵押品全部 → C 起步**，因为一旦清算就不可逆。
- **衍生品 / 永续**：查行情 / 资金费率 → A。**任何开仓 / 加杠杆都直接进 D** — 这一档我个人在产品层面不会让 Agent 自动执行，太容易因为 Oracle 抖动炸掉。
- **稳定币**：链上余额 → A。1:1 mint/redeem 链下走的，不在 Agent 自动化范围。

这张表想清楚之后我对 Day 9 那条 "Intent 边界内自动 / 边界外人工" 的边界有了具体落点 — **DeFi 里"边界"的位置就是 B 和 C 之间这条线**。B 用 Session Key 兜，C 起强制人工。

### 我对 AI + DeFi 的认知修正（今天的 1 处）

#### 修正：「Agent + DeFi」不是一个动作，是一组**风险分档**的动作

之前我会笼统说"让 Agent 帮我管 DeFi 仓位"。今天这张 4 档表把这句话拆开 — A 档完全自动没问题（甚至可以做成 always-on dashboard），B 档可以用 Session Key + Policy 自动，C 档每一笔都要人，D 档我直接不放进 Agent 边界。这条认知最大的副产品是**"哪些 DeFi 动作适合做成 Agent feature"** 也变得能讨论：A + B 是能上的，C 是 demo 友好但产品上要谨慎，D 是反面教材。

### Issue #2：Bundler 不稳定时，Agent 应该退化到什么行为？

```bash
gh issue create \
  --title "Q: ERC-4337 Bundler 不稳定时，Agent 任务的 fallback 策略" \
  --label "question,week-2,account-abstraction,agent-design" \
  --body "上下文：Day 11 读 Account Abstraction 一节时注意到 Handbook 提到'Bundler 不稳定，用户操作就可能卡住'。这件事在普通钱包用户场景大不了让用户重试，但在 Agent 自动执行场景下成本变得很真实 — 一个'要扣预算才能继续'的任务遇到 UserOp 卡住，Agent 该怎么办？

候选策略：

A. 等：保持 UserOp pending，超过 N 分钟后报警给用户。
B. fallback：自动切到非 4337 路径（如直接 EOA 签名）— 但这意味着 Smart Account 的额度护栏被绕开，**不可取**。
C. 挂起任务：把整个任务停在'等待用户确认是否切换路径'状态，把 fallback 选择交回用户。
D. 多 Bundler 故障转移：同一个 UserOp 同时投递到 N 个 Bundler。

具体问：

1. 生产环境里大家通常做哪种？还是有更成熟的 D' 方案？
2. fallback 是否应该成为 Intent / Policy 的一个字段（'本任务允许 / 不允许走 Bundler 之外路径'）？

这个问题影响产品的可用性承诺，先开 issue 跟踪。"
```

### Issue #3：Simulation 在 Agent 自动调用场景下怎么做才不浪费 gas？

```bash
gh issue create \
  --title "Q: Agent 自动调用场景下，Simulation 的成本控制策略" \
  --label "question,week-2,agent-design" \
  --body "上下文：Day 11 把 Simulation 列为 Agent Wallet 9 节点之一。用户视角的 simulation 用来'让人看懂这笔交易会失去什么 / 得到什么'，逻辑很清楚 — 每次签名前跑一次。

但在 Agent 自动调用（B 档：Session Key 内的小额执行）场景下，'每次调用前都跑一次 simulation' 可能成本失控（节点 RPC / Tenderly 调用费 + 自己服务的 CPU）。

候选策略：

A. 全量 simulate：每次都跑，最安全，最贵。
B. 阈值采样：'金额 ≥ X' 或 '资产变更 ≥ Y%' 才跑。
C. 关键路径采样：只在白名单合约的特定方法（如 approve / 跨链 / 借贷开仓）跑。
```
