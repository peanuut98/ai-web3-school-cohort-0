## 关键概念笔记 · Part 2：Solidity 三组原子 — 从「调用方」切换到「合约作者」视角

Week 2 / Day 8 / 9 读 Smart Contract 章节时，我是「调用方」视角 — 关心 ABI / 函数签名 / 怎么调，不关心源码长什么样。Week 4 进 Solidity 章节是切到「合约作者」视角 — 同一个 `function increment() public`，今天要能判断「为什么这样写」。

下面三组原子是 Day 14 部署 Counter 合约时已经接触过、但当时只当语法接受、没追问设计意图的部分。

### 原子组 1：Function Visibility（函数可见性，4 个）

| 关键字 | 谁能调 | 在哪用 | 一句话核心 |
| --- | --- | --- | --- |
| `public` | 合约外 + 合约内 + 子合约 | 默认对外暴露的函数 | 自动生成 ABI 入口 + 内部能直接调 |
| `external` | 合约外 + 子合约（**不能**合约内直接调，要 `this.f()`）| 只对外暴露的函数 | 比 `public` 省 gas（参数走 calldata，不复制到 memory）|
| `internal` | 合约内 + 子合约 | 只想给本合约 + 继承者用 | 不进 ABI，外部调不到 |
| `private` | 合约内 | 只想给本合约用 | 比 `internal` 严格一档；**注意：链上数据是公开的**，private 只是限制调用，不是隐藏数据 |

**为什么这样设计的判断**（向我妈解释体）：

> "智能合约里每个函数都得明说『谁能调』— 不像普通编程语言里很多函数默认所有人可以用。原因是合约是公开的、所有人能看到，所以**不写明白等于把家门钥匙挂外面**。这 4 档可见性是合约作者给函数挂的不同档锁。"

**回看 Day 14 那行 `function increment() public`**：

- 当时 Remix 模板就是 `public`，我没多想就部了。
- 今天判断：**这里其实更应该用 `external`** — Counter 的 `increment` 没有任何合约内调用它的地方（合约里也只有这一个函数），用 `external` 能省一点 gas。`public` 没错但不是最优。
- 这条对 Day 19 选型 A 方案（Counter 合约升级）有具体指引：升级版的 `increment` 应该改成 `external`，这是一个具体的小优化。

### 原子组 2：State Mutability（状态可变性，3 档 + 默认）

| 关键字 | 含义 | 调用方代价 | 一句话核心 |
| --- | --- | --- | --- |
| `view` | 只读链上状态，不修改 | **0 gas**（外部 read-only call）| 读余额 / 读 count 这种就用 view |
| `pure` | 不读不写链上状态 — 纯函数，只用入参算 | 0 gas | 数学计算 / 哈希 / 字符串处理 |
| `payable` | 这个函数能收 ETH | 调用方付 gas + 可附带 ETH | 不加 `payable` 函数就不能收 ETH（会 revert）|
| （默认：可写）| 修改链上状态 | 调用方付 gas | Counter 的 `increment` 就是这一档（修改 count 状态） |

**为什么这样设计的判断**：

> "合约函数对链上世界做什么 — 是只看不动、不看不动、要收钱、还是要改世界 — 必须在函数签名里说清楚。原因是**调用方根据这个决定要不要付 gas、要不要发交易**。一个 `view` 函数我可以免费读 1000 次，一个普通函数读一次就要 gas。"

**回看 Day 14 那个 Counter 的 `count()` getter**：

- Solidity 里 `uint public count;` 自动生成的 getter 是 `function count() external view returns (uint)`。
- 这就是为什么我用 ethers.js 读 count 不用付 gas — 因为是 view 调用，本质上是问 RPC 节点本地状态，不上链。
- 这条把 Week 3 的 ethers.js 实操（Day 18 读余额也是这个机制）跟今天的 Solidity 语法对应起来了 — `provider.getBalance` 在底层就相当于调一个 view 函数。

### 原子组 3：Modifier（修饰符）

Modifier 是合约作者给函数挂的「**前置检查**」— 函数体执行前先跑一段判断逻辑，不通过就 revert。最常见的就是 `onlyOwner`：

```solidity
modifier onlyOwner() {
    require(msg.sender == owner, "Not owner");
    _;  // 这一行表示「执行被修饰的函数体」
}

function dangerousAction() public onlyOwner {
    // 只有 owner 能进这里
}
```

**为什么这样设计的判断**：

> "合约函数经常要做同样的『前置检查』— 是不是 owner / 没过期 / 余额够不够。如果每个函数开头复制一遍 `require(...)`，代码乱、容易漏、改一处忘一处。modifier 把这种检查抽成一个可复用的标签，挂在函数签名上 — 一眼能看出这个函数有什么前置约束，改一处所有用它的函数都更新。"

**对应 Day 19 选型 A 方案的两个修饰符需求**：

| 升级目标 | 用 modifier 怎么写 |
| --- | --- |
| 加 cooldown 限制（10 秒只能 increment 一次）| `modifier cooldown() { require(block.timestamp >= lastCall + 10, "..."); _; lastCall = block.timestamp; }` |
| 加 owner 检查（只有 owner 能 increment）| `modifier onlyOwner() { require(msg.sender == owner, "..."); _; }` 然后 `function increment() external onlyOwner cooldown { ... }` |

这跟 Day 19 那条「Counter 合约升级 = cooldown + owner 检查」选型对得上 — 当时只是写了一句话方向，今天有了具体的语法落地。

## 关键概念笔记 · Part 3：OpenZeppelin Ownable.sol 速读

OpenZeppelin 是 Solidity 生态最常用的合约库，里面 Ownable.sol 就是工业级的 `onlyOwner` 实现。今天打开看一眼，跟我自己上面写的简化版比一比：

### 跟我简化版的差别（核心 3 点）

| 维度 | 我上面的简化版 | OZ Ownable.sol | 为什么 OZ 这样写 |
| --- | --- | --- | --- |
| owner 初始化 | 没说 | 构造函数显式接受 `initialOwner`（v5 之后强制）| 防止部署者忘了设 owner，或者用 `address(0)` 误设成无主合约 |
| 权限转移 | 没说 | `transferOwnership` + `acceptOwnership` 两步走（OwnableUpgradeable / Ownable2Step）| 一步转移如果填错地址，合约就永久失主了；两步走必须新 owner 主动接受才生效 |
| 错误处理 | `require(..., "Not owner")` 字符串 | `revert OwnableUnauthorizedAccount(account)` 自定义 error | 自定义 error 比 string 省 gas + 更结构化（前端能解析）|

### 一条收获

> **「自己能写出来的简化版」和「工业级实现」的差距，往往不在『核心逻辑』，在『失败模式的兜底』。**

我上面的简化 modifier 在 happy path 上完全正确，但 OZ 那 3 个加固都是处理「人会犯错」的场景：忘了设 owner / 转错地址 / 错误信息不结构化。这条对将来 Hackathon 写合约非常重要 — **能跑通 ≠ 能上链**，这中间的所有差距都在「失败模式的兜底」上。

这也跟 Week 3 / Day 17 那条判断（「框架的价值在『默认值取舍』不在『功能列表』」）同构 — OZ 的价值不是「写好了一个 onlyOwner」，是「在你想不到的失败模式上替你选好了默认行为」。
