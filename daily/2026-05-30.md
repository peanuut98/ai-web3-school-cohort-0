## 关键概念笔记 · Part 1：Indexing（The Graph）

来源：[Handbook · Indexing](https://aiweb3.school/zh/handbook/web3/indexing/)

第一性原理（用我自己的话整理）：

> 链是**追加日志**，不是数据库。直接 RPC 查"过去 30 天内某地址的所有 Transfer 事件"会把节点榨干 — 必须有人把日志重放、解码、按"应用语义"建索引，再用查询接口暴露出来。

核心定位：The Graph 把"全节点重放 + 解码 + 索引 + 查询"打包成一个**去中心化 GraphQL 后端**。subgraph 是"把一段合约事件流转成应用模型"的脚本，indexer 是跑这段脚本的节点，curator/delegator 是给好 subgraph 加权的人。

### 三件 The Graph 真正解决的事（每件一句话）

| 问题 | RPC 直查会怎样 | The Graph 的解法 |
| --- | --- | --- |
| **历史聚合查询** | 需要遍历从 genesis 到现在所有区块，节点会拒绝 / 超时 | 后台跑一遍把事件按"应用模型"展平进数据库 |
| **多合约关联** | 客户端要发 N 次 `eth_call` 拼数据，延迟和带宽都炸 | 一次 GraphQL 查询拿一组关联实体（user / position / pool 一起返） |
| **跨索引去中心化信任** | 自建 indexer 等于一家中心化数据库 | 多 indexer 跑同一份 subgraph，证明数据一致性，搭上经济激励 |

### 三个常见误解（我自己之前踩的）

1. **「The Graph = 链上数据」** — 不是。**The Graph 是链上数据的索引和查询层**，原始数据还在链上；subgraph 只是把"日志 → 实体"这条解读规则写明白。
2. **「subgraph 永远准」** — 不是。subgraph 是开发者写的代码，**写错就索引错**；它和合约是"两份独立逻辑"，合约升级 / 事件签名变更 / 跨合约关联出 bug，subgraph 都可能漂走。
3. **「能用 The Graph 做实时」** — 视场景。subgraph 有"指针追上链头"的延迟（通常几秒到几十秒），高频交易 / 清算监控这种需要 sub-second，The Graph 不是首选。

### 关键直觉：GraphQL ≈ 链上版"tool 查询接口"

把 Day 10 那条「ABI ≈ Agent tool schema」接过来 — Day 10 讲的是**调用**接口，今天讲的是**查询**接口。两条接到一起：

| 维度 | Agent tool / API | 链上等价物 |
| --- | --- | --- |
| 写入接口 | API endpoint + JSON body | 合约地址 + ABI + calldata（Day 10） |
| 读取接口 | REST GET / GraphQL query | The Graph subgraph + GraphQL（今天） |
| 接口契约 | OpenAPI / MCP tool schema | ABI + subgraph schema |
| 实时性 | API 给到多新就多新 | RPC 实时 / Indexer 有延迟 |

Agent 调链上数据时，**写走 RPC + 钱包，读走 Indexer 更省**。这条分工不是优化，是默认架构。

## 关键概念笔记 · Part 2：Oracle（Chainlink）

来源：[Handbook · Oracle](https://aiweb3.school/zh/handbook/web3/oracle/)

第一性原理（用我自己的话整理）：

> 智能合约**只看得到链上状态**。要让合约根据"现在 ETH 多少美元 / 这个 API 返回了什么 / 比赛谁赢了"这种链下事实做决策，必须有人把这件事**可信地写到链上**。这条桥就是 Oracle。

核心定位：Oracle 不是"链上数据库"，是**信任假设的搬运者** — 它把"我在链下看到了 X"这件事，用一组节点 + 共识 + 经济激励的形式翻译成"链上 X = 这个值"。Oracle 错了，所有读它的合约一起错。

### 四类 Oracle（每类一句话讲清）

| 类型 | 一句话 | 锚点项目 | 它最容易出问题的方式 |
| --- | --- | --- | --- |
| **Price Feed** | 多源聚合后把"标的资产价格"周期性写到链上 | Chainlink Data Feeds / Pyth | 心跳延迟 / 单源被砸 / 聚合权重失衡 |
| **General Compute / Functions** | 让合约请求一段链下计算并把结果上链 | Chainlink Functions | 计算节点串通 / 请求路径被改 |
| **Randomness（VRF）** | 给链上提供可证明的随机数 | Chainlink VRF | 抽奖 / NFT 发行被预测 |
| **Cross-chain Messaging** | 把"另一条链发生了 X"上到本链 | Chainlink CCIP / LayerZero / Wormhole | 桥被攻破 = 所有依赖该桥的合约同时风险 |

### 三个真实事故反例（DeFi 史上 Oracle 风险的具体形态）

不举具体协议名，只讲事故形态 — 因为下次同型事故还会按这几条复演：

1. **闪电贷打偏单源价格**：DEX 内部价格被一笔大单砸偏 → 借贷协议把这条 DEX 价格当唯一 Price Feed → 攻击者用偏价借出超额抵押。**教训**：Price Feed 不能只信单一来源。
2. **Oracle 心跳跑不赢市场闪崩**：标的资产 30 秒内闪崩 50% → 链上价格还停在崩前那次 update → 应该被清算的仓位"看起来"健康 → 协议穿仓。**教训**：心跳间隔 + 偏离阈值要按场景调，越敏感市场要越快。
3. **跨链桥被攻击导致跨链 Oracle 全错**：A 链发生的"我转走了 100 USDC"事件被伪造的桥消息搬到 B 链 → B 链合约误以为收到资金 → 释放等额代币给攻击者。**教训**：跨链消息是 Oracle 的子集，桥的安全模型 = 所有依赖它的合约的安全上界。

### 关键直觉：Oracle 不是"数据源"，是**信任假设**

读完之后我把 Oracle 这件事压成一句话：**"用 X 这个 Oracle = 接受 X 的失败模型作为我自己的失败模型上界。"**

这条直觉对接 Day 12 那张 4 档分层表特别要紧 —

- 借贷开仓（C 档）依赖 Price Feed → 那么 **Price Feed 的失败 = 我开仓时无法控的风险**。
- 跨链桥小额转移（C 档）依赖跨链 Oracle → 那么 **桥被攻破 = 我那笔小额连同别人的大额一起没了**。

Agent 设计时，"我用了哪些 Oracle"应该和"我接了哪些合约"一样显式记录，列在 Policy 里。

### 这张图带出来的 3 件事

1. **AI 侧只做"生成"，所有"是否允许"由 Smart Account 一侧的确定性规则承担** — 这是 Day 11 那句"生成是概率，拒绝是确定性"在全景图里的位置。
2. **写和读是两条独立通道**：写走 RPC + 钱包签名（不可逆 + 全网公开），读走 Indexer + GraphQL（便宜 + 可缓存）。Agent 应当默认用读通道做决策依据，写通道只在 Intent 边界内执行。
3. **整张图的"信任地基"是 Oracle**：DeFi 合约的清算、价格、跨链消息全部依赖 Oracle。Oracle 错 → 这张图最底层一起错 → Smart Account 的护栏没用，因为护栏只检查"调了允许的方法 + 在金额范围内"，不检查"价格是不是真的"。

### 我对 AI + Web3 的认知修正（今天的 1 处）

#### 修正：Agent 的"风险护栏"必须包含一层"Oracle 假设审计"

之前 Day 11 / Day 12 整理 Agent Wallet 9 节点 + DeFi 4 档分层时，**护栏全在"动作 / 金额 / 路径"层**。今天读完 Oracle 才意识到这层不够 — 即使所有动作合规，**底层 Oracle 错了护栏也兜不住**（闪电贷打偏单源价格那种事故里，每一笔合约调用都是合规的，问题在 Oracle）。

### Counter 合约最小源码（Day 14 直接拷进 Remix）

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract Counter {
    uint256 public count;

    event Incremented(address indexed by, uint256 newCount);
    event Reset(address indexed by);

    function increment() external {
        count += 1;
        emit Incremented(msg.sender, count);
    }

    function reset() external {
        count = 0;
        emit Reset(msg.sender);
    }
}
```

这个版本故意做最小：一个状态变量、两个写函数、两个事件。明天目标只是把它部署到 Base Sepolia，调一次 `increment`，到 BaseScan 上看交易和事件 — **不是 Solidity 学习，是部署流程跑通**。
