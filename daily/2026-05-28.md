## 关键概念笔记 · Part 1：Account Abstraction（Web3 基础）

来源：[Handbook · Account Abstraction](https://aiweb3.school/zh/handbook/web3/account-abstraction/)

第一性原理（Handbook 原话）：

> 当账户本身可以编程，权限就可以从「有私钥/没私钥」变成「在什么条件下允许什么动作」。

核心定位：AA **不是「免 gas」**，是把账户控制权从「单一私钥」扩展成「可编程规则」。这一点对我特别要紧 — 之前我一直默认 AA = gasless，今天才意识到 gasless 只是 Paymaster 的一个副产品。

### 五个零件（每个一句话讲清）

| 零件 | 一句话 | 关键边界 |
| --- | --- | --- |
| **ERC-4337** | 不直接发普通交易，而是发 `UserOperation`，由 Bundler 收集后提交到 `EntryPoint`，再由 EntryPoint 调用智能账户做验证和执行 | 是 alt-mempool + 账户合约的标准接线，**不需要修改以太坊协议**就能跑 |
| **Smart Account** | 由合约控制的账户，可以把多签 / session key / 限额 / 自动恢复 / 批量执行写进账户逻辑 | 灵活性提升的代价是**合约本身的攻击面**（bug / 模块权限 / 升级逻辑 / 外部依赖） |
| **Bundler** | 收集 UserOp，先做模拟验证再提交到 EntryPoint，相当于「面向 UserOp 的打包服务」 | 是**基础设施依赖** — Bundler 不稳定，用户操作就卡住；模拟不充分，失败交易留下成本 |
| **Paymaster** | 第三方为用户操作付 gas，或允许用户用非原生资产付 gas | **不是免费午餐**：要做风控（赞助哪些方法 / 用户额度 / 目标合约 / 防 spam / 失败成本谁担） |
| **Session Key** | 给应用 / Agent 的临时权限，**不等同于用户主私钥**：限定时间、合约、方法、额度、链 | 「中间状态」 — 让低风险动作自动化，高风险动作仍要主钱包确认 |

### Handbook 给的最小实践（设计而非部署）

> 设计一个 Agent Session Key 策略：
> 1. 选一个具体场景（例：每小时最多再平衡一次测试网小额资产）
> 2. 写清楚允许调用的合约地址和方法
> 3. 设额度、过期时间、链 ID、最大交易次数
> 4. 写出哪些动作必须回到用户钱包确认
> 5. 说明如何撤销 session key，如何审计它执行过什么
>
> 重点不是马上部署完整 AA 钱包，而是**学会把权限从「全部允许」拆成可验证规则**。

我今天就按这个最小实践给自己出了一道题（见下文「最小实践草稿」）。

## 关键概念笔记 · Part 2：Agent Wallet（AI × Web3 Bridge）

来源：[Handbook · Agent Wallet](https://aiweb3.school/zh/handbook/bridge/agent-wallet/)

这一节比 AA 那一节更直接打到「Agent 上链」这个交叉问题。第一性原理：

> 控制权不能交给一个概率系统。Agent 只能拿到**可验证、可限制、可撤销**的行动空间。

三条衍生原则：

1. **Agent 不直接控制主资产** — 模型可以生成交易草稿，但不能拿主私钥；高风险动作要回到人确认。
2. **钱包不是签名按钮，是权限系统** — 表达时间、金额、合约、方法、资产、接收方、撤销条件。
3. **自动化必须绑定撤销能力** — 用户随时能看到 Agent 拿了什么权限、做过什么、还能做什么、在哪里关掉它。

### 9 个知识节点（每个一句话）

| 节点 | 一句话 |
| --- | --- |
| **AA Wallet** | 让账户本身能表达规则，而不是被一把私钥定死 |
| **Smart Account** | 「带规则的钱包账户」— 谁能动 / 哪些操作要确认 / 哪些合约可以调 / 单日上限 / 何时失效 / 如何暂停 |
| **Safe** | 多签 + 模块化的智能账户基础设施，适合团队 / DAO / 高价值账户「把执行权拆开」 |
| **Session Key** | 临时、有限权限的小钥匙：时间 / 合约 / 方法 / 单笔额度 / 总额度 / 资产类型 / 是否可外转 / 是否可撤销 |
| **Policy** | 把「Agent 可以做什么」写成**系统能检查**的条件，不是法律条款 |
| **Guard** | 执行前的**确定性拦截层**：地址白名单 / 方法白名单 / 金额上限 / 授权异常 / 模拟结果 / 市场状态 |
| **Simulation** | 签名 / 广播前预演交易：人能看懂的「会失去什么 / 得到什么 / 哪些授权变了 / 失败成本」 |
| **Revocation** | 撤销不只发生在用户主动点击 — session key 到期、额度用完、连续失败、目标异常、行为偏离都该自动收紧 |
| **Human Check** | 不是事事手动签 — **分层**：低风险自动 / 中风险模拟 + 确认 / 高风险明确展示影响 / 超 policy 直接拒绝 |

### 两条特别戳到我的产品原则

> 所有自动化权限，都应该有一个**用户看得见的关闭入口**。

> Agent 可以负责**生成**候选动作，Guard 必须负责**拒绝**越界动作。

第二条把「AI 做什么 / Web3 做什么」的边界讲得最干净 — 生成是概率，拒绝是确定性，两件事不能由同一个组件做。

## 我对 AI + Web3 的认知修正（今天的 3 处）

按 learning-plan 的要求，反过来用今天的第一性原理检查我之前的直觉，找出 3 处需要修正的地方：

### 修正 1：AA 的关键词不是「智能账户」，是「可编程规则」

之前我一直把 AA 当成「让钱包变得更智能」 — 多签、社交恢复、批量交易这些功能堆在一起。今天意识到这些功能都是结果，不是核心。**核心是把账户验证和执行逻辑从协议层固定的 EOA 解法里释放出来**，让账户能用合约表达「在什么条件下允许什么动作」。这条认知差对后面要做的任何「Agent 上链执行」类项目都很关键 — 如果只停在「AA = 钱包功能升级」视角，会去找现成钱包；但实际要做的是设计**规则**，不是挑钱包。

### 修正 2：AA 的核心不是 gasless

之前我对 AA 的理解大概率会停在「让用户不用 ETH 也能交易」 — 因为社交媒体上 onboarding demo 都拿 gasless 当卖点。今天 Handbook 那句**「Account Abstraction 的核心不是『免 gas』，而是把账户控制权从单一私钥扩展成可编程规则」**直接把这个理解推翻。Paymaster 只是 AA 的一个零件，gasless 只是 Paymaster 的一个用途。**真正解锁的是 Smart Account + Session Key 这一条权限轴**，gas 抽象只是顺带产物。

### 修正 3：「给 Agent 权限」不是产品决定，是一组工程规则

之前我会模糊地说「Agent 在 Intent 边界内自动 / 边界外人工」 — 但「边界」是什么没说清。今天读 Agent Wallet 那一节，「边界」被拆成 8 个具体节点：**policy 写明白 / guard 当场拒绝 / simulation 提前预演 / revocation 随时收回 / human check 分层确认 / session key 表达限制 / smart account 承载规则 / safe 拆开多人执行权**。少一个节点，「边界」就只是一句口号。这条认知更新最直接的影响是 — 后面遇到「让 Agent 做 X」的产品想法，先逐项过一遍这 8 个节点，缺什么补什么，而不是先讨论 UX。

## 最小实践草稿：一个 Agent Session Key 策略

按 Handbook 的「最小实践」要求，给自己出一道题。**这是设计草稿，不部署。**

**场景**：用户允许 Agent 在一天内最多花 5 USDC，用来支付某个白名单 API 的调用费。

**Session Key 限制**：

| 维度 | 取值 |
| --- | --- |
| 有效时间 | 24 小时（从签发起算） |
| 链 ID | Base Sepolia 测试网（链 ID 84532） |
| 允许调用的合约 | USDC 合约地址（ERC-20）+ API 服务方收款合约地址（白名单） |
| 允许调用的方法 | `transfer(address,uint256)` 单一方法，**不允许 `approve` 任意额度** |
| 单笔上限 | 1 USDC |
| 总额度 | 5 USDC / 24h |
| 接收方 | 仅限白名单地址（不允许任意外转） |

**必须回到主钱包确认的动作**：

- session key 签发本身（这一签是「给 Agent 多少权限」的根契约）
- 修改白名单（增加 / 删除收款方）
- 调高额度
- 单笔 ≥ 1 USDC（超过 session key 的单笔上限）
- 提前撤销 session key

**撤销路径**：

- 用户在钱包 UI 一键 revoke
- 系统自动收紧：连续失败 ≥ 2 次 / 目标地址检测异常 / 24 小时到期
- session key 总额度用完后**不自动续期**，必须用户重新授权

**审计入口**：

- 链上：所有 `transfer` 事件可在区块浏览器查询
- 链下：Agent 自己的 reasoning + 调用日志单独留存（链上看不到「为什么调用」）

这道题做完最直接的体感是 — **「session key 限制」字段越具体，「Agent 能做什么」越可控**。模糊的「帮我低风险操作一下」根本写不进合约。

## 今天遇到的疑问（准备开 issue）

1. **Session Key 的「单笔额度」和「总额度」在合约里通常怎么实现？** 是在 Smart Account 验证逻辑里维护一个 `mapping(sessionKey => spent)`，还是放在外部 Roles Modifier？两种实现的 gas 成本和升级灵活性怎么权衡？
2. **Bundler 不稳定时 Agent 该退化到什么行为？** Handbook 提到「Bundler 不稳定，用户操作就可能卡住」。如果一个需要扣预算才能继续的任务遇到 UserOp 卡住，Agent 应该等、走非 4337 fallback、还是把任务挂起让用户确认？这个 fallback 策略影响产品的可用性承诺。
3. **Simulation 在 Agent 自动调用场景下怎么做才不浪费 gas？** 用户视角的 simulation 用来展示风险，Agent 视角的 simulation 用来 self-check — 后者在每次调用前都跑会不会成本失控？还是说在策略层做采样（比如「单笔 ≥ 阈值才模拟」「资产变更超过 X% 才模拟」）？

这三个问题今天不强求答案，会开成 issue 挂在仓库里跟踪。

## 今日反思

最大的认知更新：**AA 不是「钱包功能升级」，是「把账户控制权变成可编程规则」**。这条更新会改变后面所有「Agent 上链」类项目的入手方式 — 不是先挑钱包，是先设计规则。

第二条收获：**生成和拒绝必须由不同的组件来做**。Agent（概率系统）负责生成候选动作，Guard（确定性规则）负责拒绝越界动作 — 这条分工原则可以反复复用，不只在钱包场景。

第三条收获：**「自动化权限必须有一个用户看得见的关闭入口」是一条产品级硬约束**。这条原则把「权限管理」从后台配置变成 UI 一等公民 — 任何 Agent 类产品都该有「我授权给 Agent 了什么 / 在哪里关掉」这个面板。

合约我**还没自己部署过**，今天主要补的是直觉和规则设计能力，不是动手能力。这条边界自己心里要清楚。
