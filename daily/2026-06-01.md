## 关键概念笔记 · Part 1：Vibe Coding 是什么（Handbook）

来源：[Handbook · Vibe Coding](https://aiweb3.school/zh/handbook/ai/vibe-coding/)

第一性原理（用我自己的话整理）：

> Vibe Coding 不是"让 AI 替你写代码"，是**让人和 AI 共同维护一个能演进的项目** — 你负责意图、产品判断、边界规则，AI 负责把意图翻译成代码、跑测试、查文档；中间由一份**可被双方读懂的项目结构 + 上下文**做接口。

核心定位：Vibe Coding 的关键词不是"prompt 写得好"，是**"项目可以被 AI 持续看懂"**。一个适合 Vibe Coding 的项目应该满足三件事：

1. **结构清晰** — 文件夹按职责划分，AI 一眼能看懂"我现在要改的事在哪一层"。
2. **意图显式** — README / CLAUDE.md / 计划文件把"这个项目要解决什么 / 现在做到哪"写出来，AI 不用反复猜。
3. **小步反馈** — 每个动作小到能立刻跑、立刻看结果（编译过没 / 测试通没 / 部署上没），AI 能根据反馈自我修正。

### 三个我自己之前的常见误读

写下来当 anchor，下次自己跑偏的时候回看：

| 误读 | 实际 |
| --- | --- |
| "AI 写代码 = 我把需求扔过去等结果" | 实际是**人和 AI 一起把需求拆到能验证的小步**，每一步都人在闭环里看 |
| "Vibe Coding = prompt engineering" | Prompt 是入口，但**项目本身的可读性比 prompt 重要十倍** — 一个写得清楚的项目，AI 用很短的 prompt 也能干活；一个写得乱的项目，多长的 prompt 都救不回来 |
| "用 AI 写代码 = 不需要懂代码" | 不需要"裸写"，但**必须懂得读 + 懂得判断 AI 是不是在胡说**；判断力是这条路最贵的部分，不是手速 |

### 三个 Vibe Coding 落地的硬动作

读完之后我把它压成三条具体动作（我自己的语言）：

1. **每个项目根目录写一份 README + CLAUDE.md** — README 给人看，CLAUDE.md 给 AI 看（项目意图 / 命令清单 / 不要碰的文件 / 风格约定）。
2. **一开始就让 AI 看到所有"约束"** — 测试在哪、构建命令是什么、lint 规则什么样、有没有 type check。这些显式之后，AI 才会主动跑它们。
3. **把改动切到"小到一次能验证完"** — 一次只让 AI 做一件能 `npm run dev` / `pytest` / `cast call` 当场看到结果的事；越是大改一把梭越容易出问题。

### 关键直觉：Vibe Coding ≈ 给 AI 配一份"项目使用说明书"

接 Day 10 / Day 13 那条 ABI ≈ tool schema 的认知 — 链上合约用 ABI 告诉别人怎么调它，**项目用 README + CLAUDE.md + 文件结构告诉 AI 怎么改它**。两件事在结构上是对偶：

| 维度 | 链上合约 | 我的项目 |
| --- | --- | --- |
| 接口契约 | ABI | README + CLAUDE.md |
| 入口 | 函数名 + 参数 | 命令清单（dev / build / test）|
| 状态 | 链上 storage | git 仓库 + 当前分支 |
| 调用者 | EOA / 其它合约 | 我 + Claude Code / Cursor |
| 反馈 | tx receipt + Logs | 命令输出 + 测试结果 + 编译错误 |

这条直觉对我特别要紧 — 我是产品 + AI 协作开发路线的学员，**"项目是不是适合给 AI 做"取决于我把这份"使用说明书"写得好不好**，不是取决于我会不会写代码。

在 `~/agenttrace` 已经有一个 Next.js 项目（Day 1 那个）作为长期 Vibe Coding 实验场，今天**再单独起一个最小 Node 项目当 sandbox**，专门用来对比 Cursor / Claude Code 工作流，不和正式项目混：


跑通之后写一份 `CLAUDE.md` 说明这是一个 sandbox（"用来对比工具，不发布，不放真实 secret"），让任何后续 AI 进来都知道这个文件夹的边界。

### 第 3 步：在 Cursor 里跑一遍同样的事

打开 Cursor → File → Open → 选 `~/vibe-sandbox`。

按 `Cmd+L`（Cursor 的 chat 入口）开聊天，提一个最小请求：

> "把 index.js 改成接受一个命令行参数 name，输出 'hello, <name> from vibe coding'。如果没传 name 就默认输出 'hello, world from vibe coding'。改完跑一次给我看输出。"

观察的事：

- Cursor 会不会主动看 `package.json` 判断这是 ESM 还是 CommonJS？
- 它写完之后会不会**自动跑** `node index.js Alice` 给我看输出？
- 如果我说"现在加一个测试"，它选什么测试框架？会不会装新依赖？

### 第 4 步：在 Claude Code 里跑一遍同样的事（对比）

打开 Terminal → `cd ~/vibe-sandbox && claude`（Claude Code CLI）→ 一样的请求扔进去 → 观察同样的几件事。

### Cursor vs Claude Code 一句话对比（今天初步印象）

| 维度 | Cursor | Claude Code |
| --- | --- | --- |
| 入口形态 | IDE（VSCode fork）+ chat 边栏 | 终端 CLI |
| 我的代码视角 | 编辑器里**直接看 diff**，可以一行行 accept / reject | 终端里看 patch，accept 后落到磁盘 |
| 适合的事 | 写新组件 / 调 UI / 看着 diff 微调 | 跨多文件批处理 / 跑命令 / 写脚本 |
| 我作为非传统开发者的体感 | "**所见即所得**"，鼠标 / 键盘并用，门槛低 | "**讲清楚意图就行**"，纯文字驱动，需要更结构化的描述 |
| 上下文管理 | 默认 IDE 看到的工作区 | 我手动告诉它读哪个文件（@文件名）|

- **Claude Code**：跨文件批处理 / 跑命令 / 写脚本 / 整理仓库结构 / 写 README。
