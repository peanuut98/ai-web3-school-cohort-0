## 关键概念笔记 · Part 1：MCP 是什么（Handbook）

来源：[Handbook · 模型上下文协议（MCP）](https://aiweb3.school/zh/handbook/ai/mcp/)

第一性原理（Handbook 原话）：

> **模型不应该直接拥有世界；它应该通过明确协议访问被授权的上下文和工具。**

核心定位：MCP 解决的不是「模型会不会更聪明」，是**「模型如何以可描述、可复用、可管理的方式使用外部能力」**。这一句对我特别要紧 — 之前我一直默认 MCP = 「让 AI 能多接几个工具」，今天才意识到 MCP 不是接得**多**，是接得**规范**：能力可被描述、可被发现、可被限制、可被审计。

### 四个零件（每个一句话讲清）

| 零件 | 一句话 | 关键边界 |
| --- | --- | --- |
| **Server** | 提供能力的一侧 — 暴露 resources / tools / prompts，让 AI 客户端读信息或调动作 | **设计重点是边界** — 暴露哪些资源 / 哪些工具有副作用 / 参数 schema / 错误返回 / 是否需要用户授权 / 日志在哪记录 |
| **Client** | 连接模型和 MCP server 的一侧（IDE / Agent runtime / 聊天客户端） | **职责是用户可见性** — 当前连了哪些 server / 模型能调哪些工具 / 调用前是否要确认 |
| **Tool Schema** | 描述工具名字、用途、参数、返回值、约束 | **schema 模糊 = 模型用错误参数填空** — 不只是字段类型，还要说明「什么时候用 / 哪些必填 / 是否改外部状态 / 失败返回什么」 |
| **Permission** | MCP 真正进产品时最容易被低估的问题 | **方便不等于安全** — 至少分：只读 vs 写入 / 当前会话 vs 长期 / 是否需用户确认 / 敏感信息访问 / 外部副作用 / 可撤销 + 可审计 |

### Handbook 原话里三条特别戳我的

1. **「工具要有 schema」** — 没有 schema，模型调用工具就会变成自然语言猜参数。
2. **「权限要在协议外也成立」** — 协议能描述能力，但真正的授权、审计、隔离仍由系统实现。这条是关键 — MCP 不替代权限系统，只是让权限系统知道有哪些能力可控。
3. **「错误要可传递」** — 工具失败 / 超时 / 权限不足必须明确返回，而不是让模型猜。

### 三个我自己之前的常见误读

| 误读 | 实际 |
| --- | --- |
| 「MCP = 让 AI 多接几个工具」 | MCP 不是接得多，是接得**规范** — 可描述、可发现、可限制、可审计 |
| 「装上 MCP 就安全了」 | 协议能描述能力，**真正的授权、审计、隔离仍要由系统实现** — MCP 不替代 permission 系统 |
| 「MCP 是聊天客户端的功能」 | MCP 是**协议**，client 只是协议消费者；同一个 server 可以被 IDE、Agent runtime、聊天客户端、本地脚本同时使用 |

## 关键概念笔记 · Part 2：Vibe Coding × MCP 的对偶（接 Day 15 那条直觉）

Day 15 我写过一张对照表：「项目用 README + CLAUDE.md + 文件结构告诉 AI 怎么改它」。今天读完 MCP 把这张表补完整 — Vibe Coding 解决的是**静态可读**，MCP 解决的是**动态可调**：

| 维度 | Vibe Coding（Day 15） | MCP（今天） |
| --- | --- | --- |
| 解决什么 | AI 怎么**看懂**我这个项目 | AI 怎么**调用**我这个项目 / 系统的能力 |
| 接口契约 | README + CLAUDE.md + 文件结构 | Tool Schema |
| 边界由谁守 | 命名 + 文件夹 + 文档约定 | Permission 系统（**协议外**） |
| 反馈来自 | 编译 / 测试 / 命令输出 | Tool 调用返回值 + 错误对象 |
| 失败模式 | 项目结构乱 → AI 用很长 prompt 也救不回来 | Schema 模糊 → 模型用错误参数填空 |

合在一起：**一个适合 AI 协作的项目 = 静态 README/CLAUDE.md/文件结构（让 AI 看懂） + 动态 MCP server（让 AI 在被授权的范围内动手）**。Day 15 那条「Vibe Coding ≈ 项目使用说明书」今天扩展成「说明书 + 一组带 schema 的可调用工具」。

## 关键概念笔记 · Part 3：MCP × Agent Wallet 的结构同构（接 Day 11）

Day 11 我整理过 Agent Wallet 9 节点（AA Wallet / Smart Account / Safe / Session Key / Policy / Guard / Simulation / Revocation / Human Check）。今天把 MCP 四零件套回去，发现两套体系**结构上是同构的** — 一个管「AI 调本地工具」，一个管「Agent 调链上动作」，但底层逻辑一样。

| 关心的事 | MCP（AI 调本地工具） | Agent Wallet（Agent 调链上动作） |
| --- | --- | --- |
| 「AI 能做什么」声明 | Tool Schema | Policy |
| 调用前能不能拦下来 | Permission（必要时弹用户确认） | Guard（确定性拦截层） |
| 调用前能不能预演 | （Handbook 没强调，但好的 server 应该提供 dry-run） | Simulation |
| 调用边界外的动作 | Permission denied 明确返回错误 | Guard 拒绝 / Human Check 接管 |
| 临时授权能否撤销 | 会话隔离 / 长期授权两种模式 | Session Key 限定时间 + 一键 revoke |
| 谁负责生成、谁负责拒绝 | 模型生成调用 / Permission 拒绝越界 | Agent 生成动作 / Guard 拒绝越界 |

**3 处结构同构**（Day 16 目标 #3）：

1. **「Schema = 让模型有谱」** — Tool Schema 之于 MCP，等同于 Policy 之于 Agent Wallet。两者都是「把『能做什么』写成系统能检查的条件，不是法律条款」。
2. **「生成和拒绝必须分离」** — MCP 的模型负责生成调用 / Permission 系统负责拒绝越界；Agent Wallet 的 Agent 负责生成候选动作 / Guard 负责拒绝越界。Day 11 那条「生成是概率，拒绝是确定性」今天再次成立。
3. **「方便不等于安全」** — Handbook 在 MCP 那节用了这五个字；Day 11 我用「自动化必须绑定撤销能力」表达过同样的意思。两个领域都强调：协议 / 钱包让事情变容易，但安全要由协议外的系统兜底。

这条对照对我做产品判断的直接价值是：**未来无论是 AI 工具集成还是链上 Agent 设计，可以用同一组检查清单**（schema / 拦截 / 预演 / 越界处理 / 撤销 / 生成-拒绝分离）。

## 今日实践 · Part 1：给 Claude Code 接最小 filesystem MCP

按 Handbook 的「最小实践」要求落地一个**只读 + 白名单 + 带日志**的最小 MCP server。

### 第 1 步：选 server

Anthropic 官方维护的 [`@modelcontextprotocol/server-filesystem`](https://github.com/modelcontextprotocol/servers/tree/main/src/filesystem) 就是 Handbook 那个最小实践的现成实现 — 暴露 `read_file` / `list_directory` / `search_files` 等只读能力，**只允许访问启动时显式声明的目录**，符合「白名单 + 只读」要求。今天先用现成的，下一步再考虑自己写一个针对学习仓库定制的 server。

### 第 2 步：白名单目录选择

我有两个候选目录：

- `~/ai-web3-school-cohort-0` — 学习仓库，**已经是 public**，里面没有任何 secret，最适合给 MCP 暴露。
- `~/agenttrace` — Day 1 起的 Next.js 项目，**有 .env.local 和 .env.example**。即便 server-filesystem 不会主动读 secret，但**默认行为是暴露目录的所有非隐藏文件**，所以这个目录不直接整体暴露 — 至少要先把白名单收窄到 `src/` 子目录。

今天选 **`~/ai-web3-school-cohort-0` 作为唯一白名单根目录**，跟 Handbook 第一性原理「通过明确协议访问**被授权的**上下文」对齐。

### 第 3 步：配置（在 Claude Code 的 MCP 配置文件里加一段）

按 Claude Code MCP docs 的格式：

```json
{
  "mcpServers": {
    "learning-repo": {
      "command": "npx",
      "args": [
        "-y",
        "@modelcontextprotocol/server-filesystem",
        "/Users/delia/ai-web3-school-cohort-0"
      ]
    }
  }
}
```

关键边界（对照 MCP 四零件）：

| 零件 | 我这次的取舍 |
| --- | --- |
| **Server** | server-filesystem，只读，白名单 = 学习仓库一个目录 |
| **Client** | Claude Code（CLI） |
| **Tool Schema** | 用 server 自带的（read_file / list_directory / search_files），第一版不自定义 |
| **Permission** | **目录白名单 + 只读** = 当前会话级权限；没有写入 / 没有外部副作用 / 没有敏感信息访问 |

### 第 4 步：跑通验证

打开 Claude Code → 启动 session → 让它做三件事验证 MCP 接通：

1. **「列出 daily 目录里的所有文件」** → 期望它通过 MCP 调 `list_directory` 拿到 Day 1 → Day 15 全部文件名（不是用 shell ls）。
2. **「读 Day 11 的笔记，告诉我其中 3 处提到 Session Key 限制维度的地方」** → 期望它通过 MCP 调 `read_file`，且**能引用具体路径**（Handbook 那条「返回结果必须带来源路径」要求）。
3. **「试着写一个文件到这个目录」** → 期望它**失败**并明确返回「权限不足 / 这个 server 是只读的」，而不是静默失败。如果能写入，说明白名单 + 只读边界没生效。

第 3 件事是关键 — Handbook 强调「错误要可传递，不能让模型猜」。这一步是整套配置的安全闸。

### 第 5 步：体感记录

跑完之后记录的 3 件事（这部分是「连接成功后回填」字段，今天打卡时 placeholder 写好，跑通之后补具体观察）：

- **MCP 调用 vs 直接 shell** 的差别 — 前者会**显式列出调用了哪个 tool / 传了什么参数**，后者一行命令就过去了。这条对应 Day 11 那条「日志记录」。
- **错误模式** — 让 Claude Code 写文件 / 读白名单外路径时，错误返回长什么样。
- **Schema 体感** — Claude Code 能不能在 chat 里告诉我「这个 server 暴露了哪些 tool / 每个 tool 接什么参数」。这是验证「工具发现可被描述」的入口。

## 今日实践 · Part 2：MCP 落地的「3 个我现在能验证的判断」

读完 + 配通之后，写下我现在能用 MCP 视角做的 3 个产品判断（Day 18 用 Claude Code 写 ethers.js 时回看，对错都记账）：

1. **`vibe-sandbox` 那个 sandbox 不该接 MCP write 权限** — 它是为了对比 Cursor / Claude Code 跑 Hello World 的，**改动应该走 IDE diff，不是走 MCP** — 因为 MCP 的写权限缺一个「让我看着每一笔改动 diff」的客户端入口。这条直接呼应 Handbook「方便不等于安全」。
2. **学习仓库的 MCP 应该一直保持只读** — 哪怕将来想让 AI 帮我写 daily note，也应该是「在终端里 Claude Code 编辑 → git diff → 我看 → commit」，**不要走 MCP 直接写**。理由：daily note 是 public 仓库，写入路径多一层 = 风险面多一层，且没有显著效率收益。
3. **AgentTrace 项目（`~/agenttrace`）的 MCP 配置必须配 sub-path 白名单** — 不能整个项目根目录暴露，至少要按 `src/` / `supabase/schema.sql` 这种粒度控制。**`.env.local` 必须明确排除**。这条是 Day 11 「policy 要写成系统能检查的条件」在工具配置层的直接落地。

## 今天遇到的疑问（候选 issue + Day 19 Round 3 feedback）

1. **Tool schema 在 server-filesystem 这种通用 server 里是预定义的，但当我自己写一个 MCP server 时，schema 应该写到什么粒度？** Handbook 说「schema 模糊 = 模型用错误参数填空」 — 但 schema 太细会让 server 变成一份长文档。有没有「最小可用 schema」的经验法则（必填字段名 / 副作用标记 / 失败返回类型？）？
2. **MCP 的「会话隔离」在 Claude Code 这种 CLI 客户端里具体怎么实现？** 一个 session 关掉后，server 是否会保留状态（例如最近读过哪些文件 / 调用过哪些 tool）？这个对应 Day 11 「自动化必须绑定撤销能力」 — 关掉客户端 = 撤销了什么？
3. **MCP 在 Web3 场景里和 ERC-4337 / Smart Account 是什么关系？** Handbook 说「MCP 不是钱包安全方案；MCP 负责工具发现 / 调用格式，Web3 账户系统负责最终权限和执行边界」。但实操上，如果一个 MCP server 暴露了「sendTransaction」这种 tool，Permission 边界到底由 MCP 守还是由钱包守？两者怎么不打架？

这 3 条今天**先记位置**，准备开 issue 跟踪 + 留到 Day 19 整理 Round 3 feedback。Day 11 留下的 3 个 issue 命令今天还没跑通（昨晚 Day 15 写打卡时就提过会跑，但 Cursor 装好之后时间块用完了）— 今天再补一次，跟新增的 3 条一起。

## 今日反思

最大的认知更新：**MCP 不是「接更多工具」，是「让工具接入变规范」**。这条对我做产品判断的直接价值不仅在 AI 工具层 — 它和 Day 11 Agent Wallet 那一套 9 节点护栏是**同一种思维方式在不同领域的两次出现**。schema / 拦截 / 预演 / 越界处理 / 撤销 / 生成-拒绝分离这 6 件事，在「AI 调本地工具」和「Agent 调链上动作」两个完全不同的场景下都成立 — 这意味着我可以用同一组检查清单对未来所有「让 AI / Agent 动手」的产品做判断。

第二条收获：**Day 15 那条「Vibe Coding ≈ 项目使用说明书」今天被补完整了** — 完整版是「说明书（让 AI 看懂） + 一组带 schema 的可调用工具（让 AI 在被授权的范围内动手）」。前者是静态的，后者是动态的；两者合起来才是一个真正适合 AI 协作的项目。

第三条收获：**「MCP 不替代权限系统」是反直觉但极其重要的一条**。我之前下意识把「装了 MCP」等同于「安全连接」 — 今天 Handbook 那句「权限要在协议外也成立」直接把这个误解修正。这条认知差对将来选 MCP server 有直接影响：**评估一个 server 不能只看它支持哪些 tool，要看它有没有清晰的权限边界 / 可撤销路径 / 调用日志**。

诚实一句：**Handbook 的「写入工具权限升级方案」我今天没设计**。最小实践要求的是「只读 server 跑通 → 再设计写入工具的权限升级方案：什么时候要用户确认、如何撤销、如何审计」。今天只跑通了只读这一半，写入那一半留到 Day 18 跟 ethers.js 实操一起做（写交易 = 真正需要权限升级的场景）。
