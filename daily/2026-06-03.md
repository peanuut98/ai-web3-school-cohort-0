## 关键概念笔记 · Part 1：Frameworks 是什么（Handbook）

来源：[Handbook · Frameworks](https://aiweb3.school/zh/handbook/ai/frameworks/)

第一性原理（用我自己的话整理）：

> **框架不是「让 AI 更聪明」，是把「AI 多步执行」从一段不可观测的黑盒变成一张可拆分、可拦截、可重放的图。**

核心定位：Frameworks 解决的不是「模型能不能做」，是**「模型做一件事需要 N 步、N 个工具、N 个失败分支时，怎么把这件事写成一段能调试 / 能审计 / 能恢复的代码」**。

接 Day 15 / 16 那条线，Frameworks 在我自己的认知图里站这个位置：

| 层 | 解决什么 | 抽象 |
| --- | --- | --- |
| **Vibe Coding（Day 15）** | AI 怎么**看懂**项目 | README + CLAUDE.md + 文件结构 |
| **MCP（Day 16）** | AI 怎么以**规范的方式调**一个工具 | Server / Client / Tool Schema / Permission |
| **Frameworks（今天）** | AI 调一次工具不够时，怎么把**多步、多工具、多状态、多失败**编排成一张可观测的图 | State / Node / Tool / Handoff / Hook / Checkpointer |
| **Agent Wallet（Day 11）** | Agent 调链上动作时，怎么把**生成-拒绝-撤销**做到位 | Policy / Guard / Simulation / Session Key / Revocation / Human Check |

四层合在一起，是一张完整的「让 AI / Agent 在我项目上动手」的工程地图。

### Frameworks 都在解决的 5 个共同问题

读完一圈之后我把 4 个框架在解决的事抽出来（不是任何一个框架的官方话术，是我自己的归纳）：

1. **状态管理** — 多步执行里上一步的输出 = 下一步的输入；状态写在哪、怎么传、怎么持久化。
2. **工具调用** — 把 MCP / 函数 / API 暴露给模型，让模型按 schema 调；调用前后能不能拦下来。
3. **流程编排** — 顺序 / 分支 / 循环 / 多 Agent 协作；什么时候模型自己决定下一步、什么时候由代码强制路由。
4. **失败恢复** — 单步失败、超时、模型胡说、工具返回错；能不能从失败那一步重放，而不是从头跑。
5. **可观测性** — 每一步调了什么、传了什么、花了多少 token、用了多少时间；调试时能不能直接看到 trace。

**这 5 件事在 4 个框架里都被处理 — 区别只在「写法多重 / 抽象高低 / 默认值偏向哪一种使用场景」**。这条认知比任何一个框架的语法都更值钱：选框架不是选「哪个最强」，是选「哪个的默认值跟我这个场景的取舍方向一致」。

### 三个我自己之前的常见误读

| 误读 | 实际 |
| --- | --- |
| 「框架越多功能越好」 | 框架的价值在**默认值取舍**，不在功能列表 — 写一段最小循环每个框架都能做，但「默认怎么写状态 / 默认怎么处理失败 / 默认要不要 checkpoint」决定它适合哪一类项目 |
| 「先选好框架再开始写 Agent」 | 反过来：**先写 50 行裸 SDK 调用，跑通再判断要不要上框架**。框架是「同一段逻辑要重复 + 要被多人维护 + 要能调试」时才赚回它的复杂度 |
| 「LangChain 落伍了 / LangGraph 一定更好」 | LangChain ≠ LangGraph 的旧版本 — LangChain 是组件库（Loaders / Splitters / Vector Stores / Output Parsers），LangGraph 是编排层；做 RAG 仍然主要用 LangChain，做有状态多步 Agent 才换 LangGraph |

## 关键概念笔记 · Part 2：四个框架一句话 + 适用场景对比表

读完 + 翻官方文档之后整理出来的对比表（**今天目标 #2**）。每一行都按「一句话 / 核心抽象 / 适合做什么 / 不适合做什么 / 我自己的取舍」5 列写。

### 表 A — 一句话定位

| 框架 | 一句话 | 核心抽象 |
| --- | --- | --- |
| **LangChain** | 模型应用的**组件库** — 加载 / 切分 / 向量化 / 检索 / 输出解析每一类都有一组现成块 | Chain / Document / Retriever / OutputParser / Memory |
| **LangGraph** | 把多步 Agent 写成**状态机** — 节点是函数 / 边是控制流 / 状态是显式对象，跑出来的是一张图 | StateGraph / Node / Edge / Checkpointer / Interrupt |
| **OpenAI Agents SDK** | **极简 multi-agent handoff** — 每个 Agent 是一个带 tools 和 instructions 的对象，主路径是 Agent A 把对话交给 Agent B | Agent / Tool / Handoff / Guardrail / Tracing |
| **Anthropic Claude Agent SDK** | 把 **Claude Code 那套本地循环**（读文件 / 跑命令 / 用 MCP / subagent）抽出来做成 SDK，给 CLI / 自动化场景用 | ClaudeAgent / Tools (built-in + MCP) / Subagents / Session / Hooks |

### 表 B — 适合做什么 / 不适合做什么

| 框架 | ✅ 适合 | ❌ 不适合 |
| --- | --- | --- |
| **LangChain** | RAG 流水线 / 文档问答 / 输出格式约束 / 跨向量库切换 / 一次性原型 | 多步分支复杂 / 需要中断点和 human-in-the-loop / 长 session 持久化 |
| **LangGraph** | 多步 Agent / 分支 + 循环 / 中断点恢复 / human-in-the-loop / 长会话状态 | 一次性脚本 / 单轮 RAG / 团队对状态机思维不熟 |
| **OpenAI Agents SDK** | 多 Agent 分工 / 客服式 handoff / 需要轻量 guardrail 拦截 / 想用 OpenAI tracing 一键看 trace | 想脱离 OpenAI 生态 / 复杂分支编排（不如 LangGraph）/ 重 RAG（不如 LangChain）|
| **Claude Agent SDK** | CLI / 终端自动化 / 需要 MCP + 文件系统 + bash 一起跑 / 把 Claude Code 工作流嵌到自己产品 | 多模型混搭（绑 Claude）/ 跨 Agent 复杂分工（不如 Agents SDK 顺）/ Web 端非 CLI 形态 |

### 表 C — 我自己的取舍（针对 AI × Web3 学习路线）

| 场景 | 我会用 | 原因 |
| --- | --- | --- |
| Day 18 写 ethers.js 读 Sepolia 余额（一段脚本） | **裸 SDK，不上框架** | 一次调用、无状态、无分支 — 上框架是负杠杆 |
| 写一个「监听 Sepolia 余额变化 → 触发 Agent 决策 → 调 DeFi 合约」的多步 Agent | **LangGraph** | 显式状态 + 中断点 + 重放，对 Agent Wallet 那套护栏天然友好 |
| 让 Claude Code 帮我整理学习仓库 / 跑命令 / 接 MCP | **Claude Agent SDK**（其实已经在用 Claude Code，等价于 SDK 默认配置） | 本地 + CLI + MCP 是 Claude Code 的母语 |
| Hackathon 里搭一个「用户问问题 → 主 Agent 路由 → 专家 Agent 回答」 | **OpenAI Agents SDK** | handoff 抽象最直接，guardrail 默认就有 |
| 给项目加一个 RAG（Handbook 全文检索） | **LangChain** | RAG 组件成熟度最高，最快跑通 |

**一句话总结**：四个框架不是替代关系，是**「一组按场景分工的工具」**，跟 Day 15 那条「Cursor 和 Claude Code 不是替代是互补」的结论是同一种结构。重要的是把场景说清楚，框架自己就显式了。

## 关键概念笔记 · Part 3：Frameworks × Agent Wallet 的结构同构（接 Day 11 / 16）

Day 11 我整理了 Agent Wallet 9 节点（AA Wallet / Smart Account / Safe / Session Key / Policy / Guard / Simulation / Revocation / Human Check）；Day 16 我把 MCP 四零件（Server / Client / Tool Schema / Permission）和这 9 节点对照，识别出 3 处结构同构。今天把 Frameworks 抽象再套一遍 — 同一组检查清单第三次出现在不同领域。

| 关心的事 | Frameworks（AI 编排） | MCP（AI 调本地工具） | Agent Wallet（Agent 调链上动作） |
| --- | --- | --- | --- |
| 「能做什么」声明 | Tool / Node / Agent definition | Tool Schema | Policy |
| 调用前能不能拦下来 | Hook / Guardrail / Interrupt | Permission（必要时弹窗）| Guard（确定性拦截）|
| 调用前能不能预演 | Dry-run mode（部分框架支持）| 看 server 是否提供 | Simulation |
| 出错怎么办 | Retry / Fallback / Checkpointer 重放 | Permission denied 明确返回 | Guard 拒绝 / Human Check 接管 |
| 临时授权能否撤销 | Session / Memory 隔离 | 会话隔离 | Session Key 限定时间 + revoke |
| 谁负责生成、谁负责拒绝 | LLM 生成下一步 / Hook 拒绝越界 | 模型生成调用 / Permission 拒绝越界 | Agent 生成动作 / Guard 拒绝越界 |
| 状态在哪 | StateGraph / Memory / Checkpointer | server 自己 + client session | Smart Account storage + Session Key 配置 |

**3 处结构同构**（今天目标 #3）：

1. **「Schema = 让模型有谱」第三次出现** — Tool Schema（MCP）= Tool Definition（Frameworks）= Policy（Agent Wallet）。三者都是「把『能做什么』写成系统能检查的条件，不是法律条款」。
2. **「生成和拒绝必须分离」** — Frameworks 的 LLM 生成下一步 / Hook 或 Guardrail 拒绝越界；MCP 的模型生成调用 / Permission 拒绝越界；Agent Wallet 的 Agent 生成动作 / Guard 拒绝越界。Day 11 那条「生成是概率，拒绝是确定性」第三次成立。
3. **「失败必须可恢复 ≠ 失败必须可重试」** — LangGraph 的 Checkpointer 让我能从失败的那一步重放（state 还在），而不是从头跑；这条对应 Agent Wallet 的「Simulation 失败要能定位到哪一步、Human Check 要能从拦截点恢复」。两个领域都强调：**失败的语义不是「这次没成」，是「下次能不能从这一步接着试 / 或确定地回滚」**。

这套对照对我做产品判断的直接价值：**未来无论是选 AI 框架、设计 Agent Wallet、写 MCP server，可以用同一组检查清单**（schema / 拦截 / 预演 / 越界处理 / 撤销 / 生成-拒绝分离 / 状态可恢复）。

## 今日实践 · Part 1：Day 18 那段 ethers.js 脚本到底要不要上框架？

今天目标 #5。提前回答这个问题，免得明天动手时纠结。

### 任务画像

> 「让 Claude Code 写一段 Node 脚本，连 Sepolia RPC，读一个地址的余额，打印出来。」

按 Frameworks 5 个共同问题逐条对照：

| 问题 | 这段脚本的答案 |
| --- | --- |
| 状态管理 | 无状态（一次性脚本） |
| 工具调用 | 一个：`provider.getBalance(address)` |
| 流程编排 | 单步 |
| 失败恢复 | 失败重试一次足够，不需要 checkpoint |
| 可观测性 | `console.log` 就够 |

5 项里 4 项是「最低档」 — **结论：不上任何框架，裸 ethers.js + Node 就行。**

这条结论直接对应今天 Part 1 那条「先写 50 行裸 SDK 跑通再判断要不要上框架」的硬动作。**框架在「同一段逻辑要重复 + 多人维护 + 要能调试」时才赚回复杂度，单段一次性脚本上框架是负杠杆。**

### 但是 — 什么时候这段脚本会变成「需要上框架」？

写下变化条件做对照（未来回看时这条比脚本本身更值钱）：

1. **变成多地址 + 多链监听**（不是一次读，而是 5 个地址 × 3 条链 + 30 秒轮询）→ 状态管理 + 失败恢复变成核心 → 上 LangGraph。
2. **加上「余额变化 → 触发某个 Agent 决策 → 调一个合约」**（写动作）→ 进入 Agent Wallet 9 节点护栏的范围 → LangGraph + 接 Smart Account / Session Key。
3. **要让一个非技术队友每天看「这周 Agent 都做了哪些决策」** → 需要 tracing → OpenAI Agents SDK 或者 LangSmith / LangFuse 加在前两个之上。

**这条直觉对我特别要紧** — 我作为产品 + AI 协作开发路线，更容易陷入的错误不是「不会用框架」，是「太早上框架」。今天写下这条，明天动手的时候就有判断依据。

## 今日实践 · Part 2：Frameworks 落地的「3 个我现在能验证的判断」

读完 + 对比之后，写下我现在能用 Frameworks 视角做的 3 个产品判断（Day 19 复盘 / Hackathon 选型时回看，对错都记账）：

1. **Hackathon 不要先选框架** — Week 7 / Day 1 写选题那一页纸时，**先写「问题 + 用户 + 最小路径」，不写技术栈**。等到 Day 3 / 4 知道「这个 Agent 要走多少步 / 要存什么状态 / 要不要 human-in-the-loop」之后再选框架，不会反过来。
2. **AgentTrace 项目（`~/agenttrace`）如果将来加 Agent 决策跟踪，框架选 LangGraph** — AgentTrace 的核心需求是「看 Agent 每一步在干什么」，跟 LangGraph 的 StateGraph + Checkpointer + 中断点天然对齐；OpenAI Agents SDK 那套 handoff 抽象不太够用（AgentTrace 要看的是单个 Agent 的内部步骤，不是 multi-agent handoff）。
3. **学习仓库不需要任何框架** — 学习仓库是 Vibe Coding × MCP 的工作场，**「让 Claude Code 帮我整理 / 跑命令 / 写 daily note」用 Claude Agent SDK 默认行为就够**，不需要在仓库里引入 LangChain / LangGraph 依赖。这条对应 Day 16 那条「学习仓库的 MCP 应该一直保持只读」 — 同一个原则：**最小依赖 + 最显式边界**。

## 今天遇到的疑问（候选 issue + Day 19 Round 3 feedback）

1. **LangGraph 的 Checkpointer 在 Web3 场景里要不要持久化到链上？** Handbook 没讲。如果一个 Agent 在 LangGraph 里跑了 10 步，第 7 步签了一笔链上交易，第 9 步失败重启 — checkpoint 文件本身是链下状态，但「第 7 步那笔 tx 已经上链了」是链上事实。两者怎么对账？是不是每次 checkpoint 应该带一个「链上 tx hash 列表」做对齐？这条直接连 Day 11 Agent Wallet 「自动化必须绑定撤销能力」。
2. **OpenAI Agents SDK 的 Guardrail 和 LangGraph 的 Interrupt，哪个更适合做「链上动作前的强制人工确认」？** Guardrail 偏校验式（pass / fail），Interrupt 偏暂停-恢复式。我的直觉是 Interrupt 更接近 Agent Wallet 的 Human Check，但还没动手验证。
3. **当一个 Agent 同时要用 MCP（调本地工具）+ Web3 钱包（签链上动作）时，框架层应该把这两件事统一成一个 Tool 抽象，还是显式分成两类？** 直觉：分成两类（链下 vs 链上），因为失败语义、撤销路径、Permission 边界完全不同；但合在一起对模型更简单（一个 tool list 就够）。这是产品取舍，不是框架技术问题，但框架默认值会强烈影响这个取舍。

这 3 条今天**先记位置**，准备开 issue 跟踪 + 留到 Day 19 整理 Round 3 feedback。Day 11 留下的 3 个 issue 还在欠账（Day 15 / 16 都提过），明天 Day 18 跑 ethers.js 时一并补，本周内必须清掉。
